## 情境

你係 BD / sales lead，每月底要 prep 一個 pipeline review meeting：

- Review 過去一個月 deal movement
- Identify 邊啲 deal at risk
- 計 forecast for next quarter
- 整理一份 deck send 上 management

呢個 prep 老實講大部分時間係 mechanical：

- HubSpot / Salesforce export pipeline data
- Pivot table 整理
- 圈出 deal at risk（stage 太耐冇 movement）
- 寫 narrative summary
- 整 deck

熟手都要 4 小時。 呢一份「summary」一年做 12 次 = 48 小時 = 一個禮拜嘅 work。 而且 30% 嘅內容（boilerplate 部分）每月一樣。

呢篇用 Claude Cowork 將 60% 嘅 mechanical work 自動化，你 final review + storytelling 嘅 30 分鐘做 last 40%。

> Caveat：Cowork 仲係 research preview。CRM data 係 sensitive 嘅 business data——setup 之前 confirm 你公司 data policy 准許 AI tool 連 CRM。 個 minimum data：你嘅 personal CRM access 同 your own pipeline。Org-wide pipeline data 可能需要 IT approval。

## 跟住做

### 1. CRM export 嘅 format prep

HubSpot / Salesforce 嘅 default export 包太多 column（50+）。 第一次 setup 前，drill 落 「我每月真正要睇嘅 column」：

最 minimum：
- Deal name
- Account name
- Stage
- Amount
- Expected close date
- Last activity date
- Deal owner（如果你 manage team）
- 任何 custom field（例：deal source、product line）

Save 一個 export view，命名「Monthly Review」。

### 2. 建立 dedicated folder

```
~/Documents/sales-review/
├── 2026-06/                   ← 呢個月嘅
│   ├── pipeline-export.csv    ← 你 manually export 落嚟
│   └── (Cowork 生成 draft)
├── templates/                 ← deck template
│   └── monthly-review.pptx
└── archive/                   ← 過往幾個月
```

### 3. 用呢個 prompt setup 流程

<PromptBlock label="完整 prompt — Setup 每月 sales pipeline automation">
我想 set up 一個每月底自動執行嘅 sales pipeline review task。流程：

**Input sources：**
1. ~/Documents/sales-review/2026-XX/pipeline-export.csv（我每月底手動由 HubSpot export 落嚟）
2. 上個月嘅 draft（archive/ 入面 latest version，用嚟做 month-over-month comparison）
3. ~/Documents/sales-review/templates/monthly-review.pptx（template）

**每月最後一日晚上 8 點自動行：**

第一步：read pipeline CSV
- Validate data quality：邊啲 column 缺、邊個 deal info 唔齊
- 標 flag：data hygiene issue（例如：某 deal 嘅 close date 已過但仲喺 active stage）

第二步：calc key metrics
- Total pipeline value（by stage）
- Weighted pipeline（用 stage default probability）
- Forecast for next 90 days
- Win rate（如果有 closed deal 數據）
- Average deal size
- Average sales cycle length

第三步：deal-level analysis
- **At-risk deals**：last_activity > 30 日、stage 唔變 > 60 日、amount 異常大或細
- **Hot deals**：close_date 落喺下個月內、stage 係 final 2 stages
- **Anomalies**：上個月仲喺 early stage、今個月跳到 closing（worth 確認 data accuracy）

第四步：month-over-month comparison
- Pipeline value change（total + by stage）
- 邊啲 deal 喺 stage 之間 moved（progressed vs slipped）
- New deals added 同 lost deals 嘅 net effect

第五步：narrative draft
寫 4 段 summary（用嚟 paste 落 deck）：
1. **Headline**：one-liner capture 今個月最重要嘅 movement
2. **Pipeline health**：total value vs target、coverage ratio、any concerning trend
3. **Key wins / loses**：揀 2-3 個 specific deal 講
4. **Next month focus**：3-5 個 specific action 老闆 / team 應該 prioritize

第六步：生成 deck
- Open monthly-review.pptx template
- Update 每個 slide 嘅 data point（charts、numbers、bullet points）
- Save as「2026-XX-pipeline-review.pptx」

第七步：email 我
- Subject：「[月份] Pipeline Review Draft Ready — X% pipeline coverage」
- Body：上面第五步嘅 narrative summary + 個 file 嘅 path + 任何 data hygiene issue

**重要 constraints：**
- 唔好 send deck 出去 ANY EXTERNAL recipient
- 唔好 modify 原 CRM data
- 任何 「至於某 deal」 specific commentary，要 source quote 出原 row
- Forecast 嘅 confidence level 一定要 standby（「±15% 視乎 X 兩個 deal close timing」）

Setup 好之後 confirm，再 schedule 落每月最後一日 8pm 行。
</PromptBlock>

### 4. 第一次 dry run

第一次跑唔好 schedule，先 dry run：「即場跑一次，showcase output，唔好 write file 落 disk」。

睇下：
- Metrics calc 啱唔啱（cross-check 一個 deal）
- At-risk filter 嘅 threshold 適唔適合你 industry
- Narrative draft 嘅 framing 啱唔啱 your style（你可以 fine-tune）
- Deck slide 嘅 layout 唔係太 generic

### 5. 每月 30 分鐘 final touch

Cowork 出 draft 之後，你做 last 40%：

- Open deck，讀 narrative，加 你 inside knowledge（為咩某 deal slow down、context 點解 win 咗）
- Review at-risk list，識別邊啲係 真 risk vs 純係 stage hygiene issue
- 加 storytelling element：1-2 個 client quote、anecdote、market context

呢個 30 分鐘係**你嘅 sales judgment** 嘅 unique contribution，AI 唔取代到。 但 mechanical 嘅 60% 自動化之後，你個 brain 留 fresh 落呢度。

## 變化例子

### 變化 1：你係 frontline sales rep，自己出 weekly own pipeline review

<PromptBlock label="變化 1 — Solo sales rep 每週版">
我係 sales rep，唔 manage team，只 own my own pipeline。

請 modify 呢個 process：
- 每週日晚 8pm run（唔係月底）
- Pipeline scope 只 own deals（filter by owner = me）
- Output：1 頁 PDF（唔需要 deck），有：
  - Pipeline snapshot
  - 5 條我下週要 follow-up 嘅 deal（連 specific action）
  - 1 個 hidden gem（你 noticed but I might have missed）
- 寄落我 email，唔需要 attach file，inline body 已 OK

呢個 process 為 sales prep weekly 1-on-1 with manager。
</PromptBlock>

### 變化 2：B2B SaaS 嘅 ARR / churn 跟蹤

<PromptBlock label="變化 2 — SaaS ARR review">
我哋係 B2B SaaS，唔係 one-off deal。
請 adapt 呢個 process：

Metrics 換做：
- ARR / MRR by segment
- New ARR added / Lost ARR
- Net revenue retention (NRR)
- Churn rate by cohort
- Expansion revenue from existing accounts

At-risk identification：
- Account engagement score drop
- Renewal coming up in 90 日 + 仲未 expansion conversation
- Support ticket spike

Forecast：
- Next quarter renewal ARR + risk-adjusted
- Expansion pipeline coverage
</PromptBlock>

### 變化 3：你 manage team，要做 team comparison

<PromptBlock label="變化 3 — Team-level dashboard">
我 manage 5 個 sales rep team。
每月底要做 team-level review：

加 features：
- Per-rep pipeline value、conversion rate、average deal size
- Identify outlier：邊個 rep performance 特別高 / 低、邊個 deal cycle 特別長 / 短
- 公平 framing：用 multi-dimensional metric，唔好用 single revenue 排 rep
- 整 anonymized version 嘅 dashboard 用嚟 share with team（避免 explicit ranking）
- 我自己 read 嘅 version 有 named breakdown

Privacy reminder：呢個 process 處理員工 performance data，唔好 leak 個別 rep 嘅 detail 畀錯人。
</PromptBlock>

## 點解 sales review 啱 Cowork

呢個 task 嘅特性：
- ✓ 重覆：12 次/年同樣 process
- ✓ Mechanical：60% 係 data manipulation
- ✓ 多 source：CRM + 上月 file + template
- ✓ 有明確 output（deck）
- ✓ Failure tolerant：draft 你 final review

Cowork 唔取代你嘅 sales judgment——**佢 free you to think about narrative + strategy**，而唔係搞 pivot table。

最後一個 reminder：**呢個 process 嘅 value 唔喺慳 4 小時**，而喺你 mental energy 留落 do 嗰 30 分鐘嘅 storytelling。 用 fresh brain 寫 narrative，比起 tired brain 寫 sloppy narrative，呢個質素差距會 trickle up management 嘅 perception。
