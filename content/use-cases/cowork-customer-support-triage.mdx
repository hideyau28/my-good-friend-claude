## 情境

你開緊間 HK SME，可能係 e-commerce、可能係教育中心、可能係 SaaS。每朝 9 點開 inbox，見到 40 個 ticket 喺度：

- 一半係**重覆 FAQ**（「你哋幾點開？」「點計運費？」「可唔可以改地址？」）
- 三成係**真係要你睇**嘅 query（產品 spec、custom 訂單）
- 兩成係**complaint 或者退款 request**（嬲緊嗰啲）
- 中間夾住兩三個**urgent**（已經買咗單，今日要到貨嗰啲）

問題唔係單一條 ticket 難覆，而係**你要花 30 分鐘揀邊條先答**。 心理成本仲貴過實際打字。

Cowork 嘅角色：每朝 8:30am 自動跑一次，幫你分晒類、draft 好 FAQ 嗰批回覆，**9 點你開 inbox 已經有個整齊嘅 dashboard，review-and-send 就走得**。

> Caveat：Cowork 仲係 research preview，要 Pro/Max plan。佢會直接 read 你嘅 inbox，**第一次跑一定要 dry run，唔好畀佢自動 send**。

## 跟住做

### 1. 先準備兩份文件：知識庫 + brand voice doc

呢一步只做一次，但係**成個系統嘅靈魂**。冇呢兩份嘢，Cowork 起稿出嚟嘅嘢會似 generic chatbot。

開兩個 .md / .txt 檔，擺喺一個 dedicated folder（例如 `~/cowork-support/`）：

- **`faq.md`** — list 出你最常收嗰 15-20 條問題同標準答案。包括：營業時間、運費、退換貨政策、付款方式、出貨時間、改地址流程等等。
- **`voice.md`** — 描述你嘅 brand 語氣。例：「廣東話書面語混合、用『你』唔用『您』、句尾偶然用『啦』『喎』、唔 sell hard、有問題先 acknowledge 情緒先解釋」。

### 2. 核心 triage + draft prompt

開 Claude Desktop → Cowork mode → 指住個 folder。一次過畀佢成個 prompt。

<PromptBlock label="完整 prompt — 每朝 8:30am 自動跑">
我係 [你嘅店 / 公司 名] 嘅客服。每朝 8:30am 請你幫我做以下嘢：

1. 連去我嘅客服 inbox（Zendesk / Help Scout / Gmail label「客服」），讀晒過去 24 小時嘅未回覆 ticket
2. 讀埋 `faq.md` 同 `voice.md`，呢兩份係你嘅 reference
3. 將每張 ticket 分做以下三類：
   - **URGENT**：客戶已付款 / 已下單，今日內要回應（例如：已出貨但未到、約咗今日交收、約咗 demo）
   - **FAQ-answerable**：問題喺 `faq.md` 入面 cover 到嘅
   - **COMPLAINT / NEEDS-HUMAN**：投訴、退款、custom 要求、情緒激動、或者 FAQ 答唔到嘅
4. 對 FAQ-answerable 嗰批，每張 ticket draft 一個回覆：
   - 跟 `voice.md` 嘅語氣
   - 引用 `faq.md` 嘅事實答案，唔好作
   - 留個 `[需店主 confirm]` tag 喺需要我 fact-check 嘅地方
5. 出一份 markdown report，subject「[Date] 客服 triage」，內容分三 section：URGENT / FAQ drafted / Complaint queue

注意：
- 唔好自動 send 任何嘢
- URGENT 嗰批唔需要 draft，淨係 flag 出嚟畀我即刻睇
- COMPLAINT 嗰批淨係列出 customer、ticket subject、初步 sentiment（嬲 / 失望 / 投訴 / 退款）——等我自己處理
- 報告 email 寄返畀我

set 好之後 confirm，schedule 落每朝 8:30am 自動跑。
</PromptBlock>

### 3. 三層 output format（呢度係核心）

你想 Cowork 出嘅報告長成噉：

> **URGENT (3)**
> - 陳小姐 #4521 — 「今日 3pm 約咗到貨點解未收到」（已出貨 2 日）
> - Ken #4533 — 「demo 預咗 10am 你哋邊個 join？」
> - 林太 #4540 — 「FedEx tracking number 唔 work」
>
> **FAQ DRAFTED (12)**
> （每張 ticket 一個 collapsible，open 入去見 draft reply）
>
> **COMPLAINT / NEEDS-HUMAN (5)**
> - Annie #4519 — 退款 request，sentiment：失望
> - …

呢個 format 嘅關鍵：**URGENT 一定要喺最上**，因為你 9am 開 inbox 第一眼要見。FAQ drafted 嗰批係「批次處理」嘅 batch，complaint 係「慢工出細貨」嘅 batch。

### 4. Daily review-and-send ritual + escalation playbook

9:00am 你坐低：

1. **5 分鐘**處理 URGENT — 即刻覆，呢啲冇得 batch
2. **15 分鐘** scan FAQ drafted batch — 每張 ticket 撳 send（或者改少少先 send）。Claude draft 嘅嘢七成可以原文用，三成要微調
3. **20 分鐘** 處理 complaint queue — 呢啲先係你真正嘅工，要諗、要打電話、要 escalate

第一個禮拜你會發現 Claude 嘅 FAQ classification 有大概 10% 出錯（例如將 complaint 當咗 FAQ）。每次出錯你**喺 `faq.md` 入面加多句**「以下情況唔屬於 FAQ：客戶用咗『投訴』『嬲』『退錢』等字眼」，準確度會升。

## 變化例子

### 變化 1：WhatsApp Business 版

如果你個客服主要喺 WhatsApp 而唔係 email，先用 [WhatsApp 客戶回覆模板](/use-cases/whatsapp-customer-reply-hk) 嗰個方法 set 好 quick reply 模板，再 hook Cowork 入嚟：

<PromptBlock label="變化 1 — WhatsApp 版">
（同主 prompt 一樣，但 input source 改成）：

1. 連去我嘅 WhatsApp Business（透過 Meta Business Suite export，或者我每朝會把過去 24 小時嘅未覆訊息 export 做 .csv 擺入呢個 folder）
2. 將每段對話分 URGENT / FAQ / COMPLAINT
3. 對 FAQ 嗰批，draft 嘅回覆要：
   - 短（WhatsApp 文化，唔好寫成 email）
   - 對應返我 WhatsApp Business quick reply 嘅 short code（例如 `/有貨` `/運費`），如果有 match 就直接喺 draft 入面寫「建議用 short code: /有貨」
4. 報告同上
</PromptBlock>

### 變化 2：雙語（中英）客服版

HK 客戶經常中英混合，甚至同一張 ticket 上半段中文下半段英文。

<PromptBlock label="變化 2 — 雙語 triage">
（同主 prompt 一樣，但加多兩條）：

- 對每張 ticket 額外標一個 `language` tag：`zh` / `en` / `mixed`
- Draft 回覆嘅語言要 mirror 客人嗰封 ticket：客人寫中文你寫中文，客人寫英文你寫英文，客人 mixed 你都 mixed
- `voice.md` 入面要包括埋英文版嘅 brand voice 描述（例如：「friendly but professional, no exclamation marks, sign off with 'Cheers'」）

如果客人嘅英文唔太啱文法，唔好 mirror 佢嘅錯，用 polished 嘅英文回覆但唔好太 formal。
</PromptBlock>

### 變化 3：Complaint sentiment score 模式

對 complaint queue 加多層分析，幫你決定先處理邊個：

<PromptBlock label="變化 3 — Sentiment score 模式">
對 COMPLAINT / NEEDS-HUMAN 嗰批，每張 ticket 加埋：

- **Sentiment score**：1-5（1 = 輕度不滿，5 = 已經話要去消委會 / Small Claims）
- **Risk flag**：如果出現「消委會」「投訴」「律師」「Trustpilot」「Google review」等關鍵字，加紅色 ⚠️
- **建議優先級**：5 分 + risk flag 嗰啲排最前
- **唔好 draft 回覆**——complaint 嘅回應要我親自諗，draft 反而會誤導我

排序：risk-flagged → score 5 → score 4 → 其餘
</PromptBlock>

## 一個 HK 客服特有嘅 caveat

HK consumer council（消委會）投訴 dynamic 同其他市場唔同：客人有時會喺 ticket 入面**直接 quote 條例**或者**話要 escalate 上消委會**。呢類 ticket Claude 識唔識認得到，**完全睇你 `faq.md` 入面有冇教佢**。

最 minimum 喺 `faq.md` 寫一段：「以下字眼出現就一定當 COMPLAINT、score 5、risk-flag：消委會、Consumer Council、Small Claims、Trustpilot、Google review、投訴信、律師信」。呢句一加，Claude 對 HK 客服場景嘅敏感度即刻上一級。

Cowork 唔係幫你**全自動回覆客戶**——係幫你**將早上嗰 30 分鐘 triage 時間，壓縮做 5 分鐘 review**。剩返嗰 25 分鐘，留返畀真正要動腦嘅 ticket。
