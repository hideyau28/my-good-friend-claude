## 情境

朝早 9:15，你坐低，打開 inbox：

> **62 unread**

你開始碌。第 14 封係老闆喺長 thread 第 7 個回覆嗰度 @ 你：「呢個你跟一跟」。第 22 封係供應商報價，要你星期五前覆。第 38 封係客最後一行寫「順便要埋上季 sales breakdown」。

你睇完，知道有嘢做。中午食飯，你已經唔記得邊三件係今日真係要動嘅。下晝 4pm 老闆 ping 你：「嗰單你跟咗未？」你心諗：「邊單呀？」

問題唔係 inbox 多，係 **要做嘅嘢散晒喺 thread 入面、冇統一一張清單**。你個腦做緊兩件事：（1）讀 email、（2）即時記住邊啲要做。第二件事，個腦做得好差。

呢篇教你設定 Cowork 每朝 8am 自動跑：掃過去 24 小時 inbox → 抽真正要做嘅嘢 → 連死線、來源 email、優先次序一齊寫入 Notion master task DB。9am 你開電腦，Notion 已經有今日要做嘅清單。

> 注意：呢個工作流會接你 inbox 同寫入 Notion。要 Pro/Max plan 加一個專用嘅 Notion DB（**唔好直接接你私人嗰個 master list**）。第一個禮拜一定有抽錯嘅情況，要逐日微調。

## 跟住做

### 1. 設定 Notion integration + Cowork email access

開始之前要備齊兩樣嘢：

**Notion 邊：**
1. 入 `notion.so/my-integrations` → 開個新 integration，叫「Cowork Inbox」
2. 攞個 integration token（`secret_xxx`）— 記低，等陣畀 Cowork
3. 喺 Notion 開個新 database，叫「Inbox Tasks」，欄位結構：
   - **Title**（text）— 要做嘅嘢嘅描述
   - **Due**（date）— Cowork 估計
   - **Source**（URL）— 原 email link（Gmail thread URL）
   - **Priority**（select：高 / 中 / 低）
   - **Status**（select：未做 / 做緊 / 搞掂）
   - **Captured**（date）— Cowork 跑嗰日
4. 撳個 DB 右上角「Share」，邀請你頭先嗰個 integration，畀「Can edit」

**Cowork 邊：**
- Claude Desktop → Cowork mode → 連你 Gmail / Outlook（OAuth）
- 連 Notion → 貼個 integration token

呢一步係苦工，但只做一次。

### 2. 核心 prompt — 點分「要做」同「淨係知會」

呢個 prompt 嘅關鍵唔係「抽晒所有嘢」，係**抽得準**。第一個禮拜寧願少抽，唔好亂抽。

<PromptBlock label="完整 prompt — 每朝 8am inbox → Notion">
我想設定每朝 8am 自動跑嘅 inbox → Notion task 工作流。

**觸發：** 每日 08:00（HKT）

**第一步：收集**
- 由我 Gmail / Outlook 攞過去 24 小時（昨日 8am 到今朝 8am）所有 email
- 排除：
  - 推銷 / newsletter（有 List-Unsubscribe 標頭嘅）
  - 我已經覆咗嘅
  - 純系統通知（GitHub PR 通知、Calendar invite、Slack digest 等）

**第二步：抽要做嘅嘢（呢度最緊要）**

**只係**將以下 email 計做有嘢要做：

A. **有人明確要我做嘢**
   - 「可唔可以幫我 review X」
   - 「請喺 X 號前覆 / 提供 / 交 X」
   - 「你嗰邊跟到 X 點？」
   - 我喺長 thread 入面被 @ + 後面有 question / request

B. **我答應過要做、而對方等緊**
   - 我之前回過「好我跟」、「let me check」、「我聽日畀你」
   - 對方今日再 ping 我（即係到期）

**唔算**要做嘅嘢（呢啲唔好放入 Notion）：
- 淨係知會我 / cc 我但冇問我嘢嘅
- Newsletter / 行業資訊
- 已經有明確負責人、唔係我嘅 thread
- 「saw your message, thanks」呢類純回應
- 純粹報告近況（除非佢結尾問我意見）

如果一封 email 有多件要做嘅嘢（罕見），拆做多個 task。

**第三步：每件要做嘅嘢整理**

每個 task 出以下欄位：

- **Title**：用動詞起句，30 字以內。例：「覆 ABC vendor 報價 — 確認 PO 數量」、「畀上季 sales breakdown Linda」
- **Due**：根據 email 內文估
  - 對方明確寫死線 → 用嗰日
  - 「ASAP」 / 「today」 → 今日
  - 「呢個禮拜」 / 「by end of week」 → 今個禮拜五
  - 老闆 / 直屬上司、冇講時間 → 兩日內
  - 同事 / 外部、冇講時間 → 三日內
  - 唔肯定 → 今日 + 5 日（畀我自己再 triage）
- **Source**：原 email 嘅 web link（Gmail 嗰個 thread URL）
- **Priority**：
  - **高**：老闆 / 直屬上司 / 客戶寄嚟 + 24 小時內到期
  - **中**：同事或外部 + 三日內到期
  - **低**：超過五日到期 / 唔急嘅內部協作
- **Captured**：今日日期

**第四步：寫入 Notion**

用 Notion API 寫入「Inbox Tasks」DB。Status 預設「未做」。

去重規律：如果 Source URL 已經喺 DB 入面（同一條 thread 之前抽過），唔好建第二個 task —— 除非新 email 真係有新嘢要做（例如對方追第二件事），噉就建新 task，同舊嗰個 link 落 source 嘅 row description。

**第五步：寄日摘要畀我**

寄 email 畀我，主旨：「[YYYY-MM-DD] 今日 inbox queue — X 個高 / Y 個中 / Z 個低」

內容：
- 高優先清單（title + due + Notion link）
- 中優先簡列
- 低優先數量（唔展開）
- 「我考慮過但冇放入嘅」：列 3-5 封我可能有意見、但喺 prompt 規律下判斷成淨係知會嘅，等我覆核

**重要限制：**
- 唔好覆任何 email、唔好封存、唔好標做已讀
- 唔好改動既有嘅 Notion task（只係 append 新嘅）
- 任何唔肯定嘅 thread，寧願標「需人手檢視」放低優先，唔好亂猜

設定好之後同我核對，再排程落每朝 8am。第一次跑用預演（顯示輸出、唔好真係寫入 Notion）。
</PromptBlock>

### 3. Notion 欄位結構 + 寫入動作

呢個 prompt 寫得長，但你細心睇，個重點係**第二步嗰張「咩唔算要做嘅嘢」嘅清單**。第一個禮拜你會發現，Cowork 抽得太鬆——你會見到一堆「同事問我意見、但其實已經 cc 咗負責人」嘅嘢入咗 Notion。逐日微調：見到一個抽錯嘅，更新返 prompt 嘅排除條件。

第三步嗰個「死線估算規律」係硬性規則，避免 Cowork 憑感覺估，要佢跟規律估。咁第一個禮拜你覆核嗰陣，可以好清楚知道：「呢個估錯日期，係因為規律寫得唔好」，唔係 Cowork「判斷飄忽」。

第五步嗰個「我考慮過但冇放入嘅」係**最重要嗰段**——你睇返嗰 3-5 封，就知 Cowork 個判斷邊度同你唔同。呢張清單就係微調 prompt 嘅原材料。

### 4. 每日覆核習慣 — 點同呢張清單共處

朝早 9am 開電腦：

1. 開 Notion「Inbox Tasks」DB，篩「Captured = 今日」加「Status = 未做」
2. 由上至下掃一次（通常 5-10 個 task）
3. 唔同意嘅，撳入去原 email link 重新讀過。判斷錯，刪咗佢
4. 同意嘅，排好你嘅執行次序（高優先同短嘅做先）
5. 每件做完撳「搞掂」

**第一個禮拜會痛苦**：你會發現 Cowork 抽錯嘅比率大約三四成。日日記低你刪咗咩、點解，星期五覆檢一次，更新 prompt 嘅排除清單。

**第二個禮拜**：抽錯跌到一兩成。你開始信佢嘅高優先判斷。

**第三個禮拜**：抽錯落到半成到一成。你朝早唔再「掃 inbox」，而係「覆核清單」。呢個係質變。

呢個工作流嘅真正回報唔係慳時間——係**將「我有冇睇漏嘢」嘅焦慮，轉移畀一個會逐封 email 認真讀過嘅系統**。你個腦終於可以唔再喺背景度做哨兵。

## 變化

### 變化 1：用 Things 3 / Todoist 取代 Notion（URL scheme）

<PromptBlock label="變化 1 — Things 3 / Todoist 取代 Notion">
我用緊 Things 3（或 Todoist）做 master task list。

修改：
- 第三步出嚟嘅每個 task，唔好寫入 Notion
- 改為生成 Things 3 URL scheme link（things:///add?title=...&when=...&notes=...）
- 或者 Todoist Quick Add API call
- 寄畀我嘅日摘要 email 入面，每個 task 後面跟住個 deep link，我撳一下就入晒去
- Source（原 email link）放入 notes 欄位

呢個方案唔需要 Notion integration，但要求你個 task app 接收得到 deep link。
</PromptBlock>

### 變化 2：Slack messages 接落同一個 pipeline

<PromptBlock label="變化 2 — Slack DM / channel mention 接同一個 pipeline">
我啲要做嘅嘢唔淨止喺 email，仲多嘅係 Slack DM 同 channel @ 我。

修改：
- 第一步收集擴大到 Slack：過去 24 小時嘅 DM + 我被 @ 嘅 channel message
- 第二步用同一套規律抽要做嘅嘢
- Source 欄位用 Slack message permalink（唔係 Gmail thread URL）
- Title 前面加「[Slack]」做識別，方便我喺 Notion view 篩選

注意：Slack 嘅雜訊比 email 高啲（standup、emoji 反應、閒聊），prompt 嘅排除清單要加：
- 純 emoji reaction
- standup / 報告近況嘅 channel（除非有人問返我嘢）
- channel announcement
</PromptBlock>

### 變化 3：WhatsApp Business inbox 都接（HK 老闆版）

<PromptBlock label="變化 3 — HK 老闆版：加埋 WhatsApp Business">
我係小店 / 中小企老闆。我要做嘅嘢一半喺 WhatsApp Business（客 / 供應商 / 同事）。

修改：
- 連 WhatsApp Business API（透過 Meta Business Suite）
- 第一步收集加：過去 24 小時嘅 WhatsApp 對話
- 第二步抽要做嘅嘢：
  - 客落單 / 改單 / 追貨 → 高優先
  - 供應商報價 / 確認 PO → 中優先
  - 員工問放假 / 問人手 → 中優先
- Source 欄位用 WhatsApp Business 入面嗰條 chat 嘅 deep link
- Title 前面加「[WA]」做識別

私隱注意：WhatsApp 內容比 email 私人，呢個工作流嘅 Notion DB 一定要設定為「只你睇」。團隊唔好分享。
</PromptBlock>

## 拆解：點解會 work，同點解會爆

跟到上面就已經用得。下面呢段係畀**想由「跑一次 demo OK」做到「跑足幾個月都信得過」**嘅人——初學者可以跳過，唔影響你跟住做。

自動化最唔老實嘅地方係：**第一次跑靚仔，唔代表第十次冇事**。呢個 inbox-to-Notion 工作流，實際會喺呢幾個位爆，你要預咗：

**1. 去重規律只認 URL，認唔到「同一件事」**
Prompt 叫 Cowork 用 Source URL 去判斷係唔係重複，但同一件事可以喺唔同 thread 出現——對方 forward 咗封 email、或者喺 reply-all 提起同一件事。Source URL 唔同，Cowork 唔知係同一件，乖乖建第二個 task。
- 會爆成點：Notion 入面出現兩三個「畀 Linda 上季 sales breakdown」，你唔知邊個算數，兩個都唔搞掂，反而搞到更亂。
- 點救：定期（每週一次）喺 Notion 搜一搜 Title 有冇明顯 duplicate，手動 merge 佢哋；或者喺 prompt 加埋「如果 Title 同現有 task 高度相似，mark 做潛在重複、唔好靜默新增」。

**2. 24 小時窗口係假設「日日準時跑」**
Prompt 叫 Cowork 抽「過去 24 小時」email。但如果某朝 Cowork 冇成功跑（斷網、token 過期、手動暫停），隔日跑嗰次仍然係抽「過去 24 小時」——夾中間嗰段 email 就永遠唔會被掃到，你完全唔知有嘢漏咗。
- 會爆成點：漏咗一封老闆 @ 你嘅 email，你唔知，任務失蹤，直到老闆再 ping 你。
- 點救：喺摘要 email 加一行「本次掃描時間範圍：YYYY-MM-DD HH:MM 至 YYYY-MM-DD HH:MM」——你每日掃一眼，見到有斷層就即刻手動補掃；Notion 入面「Captured」欄位都可以幫你發現有冇日子缺口。

**3. Notion API write 靜默失敗**
Cowork 成功抽咗 task，但寫入 Notion 嗰步因為 token 過期、DB 分享設定被改或 API 偶發錯誤而失敗，Cowork 可能只係記落 log 而唔會大聲話你知。你以為入咗 Notion，實際冇。
- 會爆成點：你朝早開 Notion 見得少，以為今日唔多嘢，其實係抽咗但冇入到。重要 task 失蹤。
- 點救：喺 prompt 第五步加「如果 Notion 寫入失敗，喺摘要 email 頂部大粗體寫『⚠ Notion 寫入失敗，以下 task 未入 DB』」，強迫 Cowork surface 寫入結果，唔好靜默吞咗錯誤。

**4. 死線估算規律係靜態嘅，但語境係動態嘅**
Prompt 寫死「老闆冇講時間 → 兩日內」。問題係：老闆問「你有冇空跟一跟呢個方向？」同老闆寫「呢個你跟一跟，我下晝要向上報」，用同一條規律估，出嚟係一樣嘅「兩日內」——但急緩差天共地。
- 會爆成點：真正急嘅嘢被估成「兩日內 中優先」，你當尋常 task 排，唔及時跟進，老闆嬲。
- 點救：規律係底線，唔係全部。摘要 email 嗰「我考慮過但冇放入嘅」清單要認真睇；同時第一個禮拜見到「死線估得怪」嗰條，即刻打開原 email 對比，記低係邊類語境估唔準，更新返 prompt。

**5. 摘要 email 睇多咗，原 email 睇少咗**
Cowork 每朝寄埋日摘要、列晒 task，好方便——但你開始信摘要多過原文。Task title 係 30 字描述，抽咗 context。你做嗰件事之前唔再入去讀原 email，靠記憶同 title 做，容易做錯方向。
- 會爆成點：「覆 ABC vendor 報價 — 確認 PO 數量」，你回一個數量，但原 email 其實係問你有冇 budget approval，兩件唔同嘅事。
- 點救：**做嗰件事之前一定撳 Source link 讀返原 email**，唔好靠 title 猜。Notion task 嘅 Source 欄位係關鍵習慣，唔係裝飾。

呢幾個位，就係「跑一次 demo、Notion 入咗幾個 task」同「跑幾個月、你真係唔再喺背景擔心漏嘢」之間嘅距離。

> Email、Slack、WhatsApp 唔係「inbox」——係「**有人對你有期望嘅暫存區**」。個腦記唔記得呢啲期望，係件碰運氣嘅事。將佢哋抽出嚟、變成一張你會打開嘅清單，先至係個腦可以投資喺高判斷力工作嘅前提。Cowork 唔係幫你慳時間，係幫你**搬走嗰個「我會唔會睇漏嘢」嘅背景程序**。
