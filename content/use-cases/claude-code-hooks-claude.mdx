## 情境

你 Claude Code 用 6 個禮拜:

- 尋日突發: Claude 改大 refactor，你手動 `git commit` — 但唔記得 run `npm run format`。CI lint fail，要 amend commit revert + format + re-push。額外 15 分鐘 hassle。

- 上禮拜: 你 testing 一個 destructive script，唔小心同意咗 Claude「run rm -rf node_modules」(intended for cleanup) — 但 Claude 跑去 `rm -rf /tmp/某 important folder`。Recovery 1 hour。

- 上個月: 你 push 一個 commit, 後來 spotted 你 import 咗 production API key 入 .env.example file。Embarrassing leak。

呢類錯誤共同 pattern: **靠手動紀律**。你個腦應 remember 每次 do X (format / verify / scan)，但個腦偶然會唔記得 = 錯誤漏出去。

**Hook** 解決: Claude Code execution lifecycle 嘅 specific event 你可以 hook 入去 — 自動 run 你想要嘅 check / action。

**3 個關鍵 event**:

- **PreToolUse**: before Claude runs any tool (read / write / bash / etc) — 可以 inspect + block
- **PostToolUse**: after tool runs — 可以 react / chain
- **Stop**: end of session — final verification / cleanup

呢篇拆 5 個必備 hooks + setup walkthrough。

## 跟住做

### 1. Hook architecture

Hook = 一個 shell script 你 register 喺 Claude Code settings.json。每次 specific event fires, Claude Code 自動 run 你 script + decide action based on script's exit code。

```
[Claude action: write file] 
   ↓
[PreToolUse hook fires] 
   ↓ (exit 0: allow / exit non-0: block)
[Tool runs] 
   ↓
[PostToolUse hook fires] 
   ↓
[Next action]
```

### 2. Setup hook 喺 settings.json

Open settings:

```bash
claude --config
# 或手動 edit ~/.claude/settings.json
```

Hook structure:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/pre-bash-safety.sh"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/post-write-format.sh"
          }
        ]
      }
    ]
  }
}
```

- `matcher`: tool name (Bash / Write / Read / Edit / etc) — hook 只係 matching tool 嗰陣 fire
- `command`: shell script path

### 3. Hook 1 — Block 破壞性命令 (safety)

Create `~/.claude/hooks/pre-bash-safety.sh`:

```bash
#!/bin/bash
# Read Claude's intended command from stdin
input=$(cat)
command=$(echo "$input" | jq -r '.command')

# 列 danger pattern
declare -a danger_patterns=(
  "rm -rf /"
  "rm -rf ~"
  "rm -rf \$HOME"
  "dd if=/dev/"
  "mkfs"
  "shutdown"
  ":(){:|:&};:"  # fork bomb
  "chmod -R 777 /"
  "git push --force.*main"
  "git push --force.*master"
)

# Check pattern
for pattern in "${danger_patterns[@]}"; do
  if echo "$command" | grep -qE "$pattern"; then
    echo "🚫 BLOCKED: command matches danger pattern '$pattern'" >&2
    exit 1
  fi
done

exit 0
```

`chmod +x ~/.claude/hooks/pre-bash-safety.sh`

Register hook (per Step 2 example)。

Effect: 如 Claude 試 `rm -rf /` (真嘅或唔小心)，hook block + Claude see error → try alternative approach。

### 4. Hook 2 — Write 完自動 format

Create `~/.claude/hooks/post-write-format.sh`:

```bash
#!/bin/bash
input=$(cat)
file_path=$(echo "$input" | jq -r '.tool_input.file_path')

# 確認係 code file
if [[ "$file_path" =~ \.(ts|tsx|js|jsx|json|md)$ ]]; then
  cd "$(dirname "$file_path")"
  # Use prettier if installed
  if command -v prettier >/dev/null; then
    prettier --write "$file_path" 2>/dev/null
  fi
fi

exit 0
```

`chmod +x` + register PostToolUse for Write matcher。

Effect: Claude write 新 file / edit existing file → prettier format 自動 run → 你永遠唔會見到 formatting lint error。

### 5. Hook 3 — Stop session 時 cleanup

Create `~/.claude/hooks/stop-cleanup.sh`:

```bash
#!/bin/bash
# Auto-cleanup .DS_Store + temp file at session end
find . -name ".DS_Store" -delete 2>/dev/null
find . -name "*.tmp" -delete 2>/dev/null

# Check for any uncommitted changes
if [ -n "$(git status --porcelain 2>/dev/null)" ]; then
  echo "⚠️  Uncommitted changes detected" >&2
fi

exit 0
```

Register as Stop hook:

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [{
          "type": "command",
          "command": "~/.claude/hooks/stop-cleanup.sh"
        }]
      }
    ]
  }
}
```

Effect: 每次你 exit Claude Code session → cleanup runs → 你 git working dir 永遠唔會留低 cruft。

### 6. Hook 4 — Pre-commit 安全 check

Create `~/.claude/hooks/pre-bash-precommit.sh`:

```bash
#!/bin/bash
input=$(cat)
command=$(echo "$input" | jq -r '.command')

# Detect git commit attempt
if [[ "$command" == "git commit"* ]]; then
  # Check for hardcoded secrets in staged files
  if git diff --staged | grep -qE '(api[_-]?key|secret|password)\s*=\s*["'"'"'][^"'"'"']{8,}'; then
    echo "⚠️  Potential secret detected in staged file" >&2
    echo "Review with: git diff --staged | grep -iE 'key|secret|password'" >&2
    exit 1
  fi
  
  # Check for debug code
  if git diff --staged | grep -qE 'console\.log\(|print\(.*DEBUG'; then
    echo "⚠️  Debug console.log/print left in staged files" >&2
  fi
fi

exit 0
```

Register PreToolUse for Bash matcher (連 safety check 一齊)。

Effect: Claude 試 `git commit` → hook check staged content → 如 detected secret → block + suggest review。

### 7. Hook 5 — Code change 完自動 test

Create `~/.claude/hooks/post-write-test.sh`:

```bash
#!/bin/bash
input=$(cat)
file_path=$(echo "$input" | jq -r '.tool_input.file_path')

# Run tests if source code change
if [[ "$file_path" =~ /src/.+\.(ts|tsx|js|jsx)$ ]]; then
  if [ -f "package.json" ]; then
    npm test --silent 2>&1 | head -20
  fi
fi

exit 0
```

Effect: Claude edit src file → test 自動 run → result 顯示畀 Claude → Claude react if fail。

⚠️ 唔啱大 test suite (慢)。對大 project，限制 specific test pattern 或者跳過。

## 變化

### 變化 1：Project-scope hook

User-level hook in `~/.claude/settings.json` 適用全部 project。Project-scope hook in `./.claude/settings.json` 只適用呢個 project:

```json
// .claude/settings.json in project root
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "./.claude/hooks/project-specific.sh"
        }]
      }
    ]
  }
}
```

Use case:
- 某 project 嚴格 lint rule
- 某 project 唔想 auto-format (legacy code)
- 某 project 具體 deploy 命令需要額外 guard

### 變化 2：Conditional hook (event filter)

進階 hook 可以 inspect event detail before action:

```bash
#!/bin/bash
input=$(cat)
file_path=$(echo "$input" | jq -r '.tool_input.file_path')

# Only fire for production code (skip test files)
if [[ "$file_path" =~ /test/ ]] || [[ "$file_path" =~ \.test\. ]]; then
  exit 0
fi

# Specific check for prod code
# ... rest of script
```

呢個 granularity 等你 hook 只觸發真正 relevant case。

### 變化 3：Hook + Skills + Commands 三者組合

3-tier customization stack: hooks + skills + slash commands。

- **Hooks**: 被動自動 (event-triggered)
- **Skills**: workflow logic (自動 detect intent)
- **Commands**: 用戶明確 trigger

Use case:
- `/deploy` command 觸發 → spawn 4 subagents (並行 checks) → 每個 subagent 寫 file → PostWrite hook 自動 format → 一切 deploy gate clean → 你 1 個 word 搞掂。

呢個 architecture 高 sophistication 但 highly leveraged daily workflow。

## 一個 mindset

Hooks 嘅深層 insight: **AI agent + human 協作嘅安全 + quality 唔應該全靠手動紀律。將煩悶嘅 guardrails 自動化。**

你個腦 capacity 應該 spend 喺 judgment (邊個 design choice 較好, 邊個 trade-off 該揀)，唔好 spend 喺 remembering (run format / run test / check secret)。Hooks 將後者 offload 落 system。

呢個 reframe: hook 唔係「micromanage Claude」，hook 係「自動化我自己嘅紀律」。Claude action OK → hook 靜靜地。Claude action 有問題 → hook 攔截 + 防止我會做嘅錯誤。

最後 reminder：

- ✅ **由 1-2 個 hook 開始** (safety + format)。每月加 1 個，當你發現重複嘅手動動作。
- ⚠️ **Hook script bug 可以 block 全部 Claude action**。隔離 test hook 至 depend on it。
- 🎯 **Share team-level hook 入 git repo**。Pre-commit secret scan 應該係 team-wide。
