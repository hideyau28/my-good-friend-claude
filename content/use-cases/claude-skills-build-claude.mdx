## 情境

你做 admin，公司每星期出 3-5 封客戶 email。每封流程一樣：

1. 你打一段 message
2. 同 Claude 講「polish 呢段 —— 要專業但唔生硬，要 HK 客感覺真誠，唔好行貨味」
3. 貼 3 個過往好嘅 example 做參考
4. 貼公司 tone 規範
5. Claude 出 3 個版本
6. 你揀一個 fine-tune

每次 5 段貼 + 揀 = 12-15 分鐘。每星期 5 次 = 75 分鐘。每年 65 個鐘貼同一啲嘢。

呢個問題同 Projects 唔同：

- Projects 解決「所有 chat 共用同一 context」
- 但你呢個情況係「同一 workflow 用喺唔同內容」—— 每次 polish 唔同 email，但流程一樣

呢個就係 **Claude Skills** 解決嘅問題。

**Skill** = 一個 markdown 文件（加可選 helper files），描述：

- 點 trigger 個 workflow
- 用咩參考 / 規則
- Output 跟咩結構

Build 完之後，Claude 喺對話入面會自動 detect 應該叫邊個 skill，跟住個 workflow 行。

呢篇 45 分鐘 build 第一個 skill，往後幾百封 email polish 都唔需要再貼規則。

⚠️ Skills 而家係 Pro / Team / Max plan 嘅功能（部分早期版本）。

## 跟住做

### 1. 認識 skill 嘅結構

每個 skill 係一個 folder，入面有：

```
my-skill/
├── SKILL.md              # 主要 instruction 文件
├── reference/
│   ├── good-examples.md  # 你過往好嘅 output（可選）
│   ├── tone-rules.md     # 詳細規則（可選）
│   └── templates/
└── scripts/              # 進階：bash / python helper（可選）
```

最簡單嘅 skill = 1 個 SKILL.md。複雜嘅 skill 加參考 files + scripts。

SKILL.md 嘅結構：

```markdown
---
name: email-polish-hk
description: Polish English email for HK client — professional but warm, avoid corporate cliche. Trigger keywords: polish, email, client message, 改 email.
---

# Email Polish (HK)

## When to use
- User pastes a draft email + asks for polish / improve / tone-adjust
- 唔好 trigger 喺 social media / 內部 message

## Step-by-step
1. 讀 draft 揾 3 樣嘢：唔清晰嘅句 / 行貨 / 過度 hedging
2. 出 2-3 個 polish 版本，每個角度唔同
3. 結尾標明邊個版本你建議 + 1 句原因

## Rules
- 唔用：「I would like to」/「Please be advised」/「Kindly」
- 改用：「Want to」/「Just wanted to flag」/「Could you」
- 短句 > 長句
- 結尾唔好「Thank you for your understanding」—— 改用「Let me know if anything」

## Reference
See `reference/good-examples.md` for 5 past polished emails.
```

呢個 frontmatter（頂嗰個 YAML block）嘅 `description` 係**關鍵** —— Claude 用呢段判斷「呢個對話應唔應該叫呢個 skill」。寫具體啲。

### 2. Build 第一個 skill

打開文字編輯器，建立 folder `email-polish-hk/`，建立 file `SKILL.md`，貼上面個 example 改成你具體情況嘅內容。

關鍵欄位：

**Name**：kebab-case，唯一識別 skill
**Description**：包括 trigger keyword + 用途講清楚。例如：「Trigger when user asks to polish / 改 / refine an English email for HK client. Skip for casual social posts.」
**When to use** section：明確界線，等 Claude 知幾時唔好叫
**Step-by-step**：你想 Claude 跟嘅程序
**Rules**：硬性規限
**Reference**：連結去 supporting files

### 3. 加參考 files

`reference/good-examples.md` 例：

```markdown
# Past polished emails (good examples)

## Example 1 — Late delivery notice
**Original**: 「I would like to inform you that the order has been delayed due to unforeseen circumstances.」
**Polished**: 「Just heads up — your order's delayed by 3 days. Quick context: factory had a power issue Tuesday. New ETA Friday. Sorry about this.」
**Why better**: 具體原因 + 具體日期 + 真誠道歉

## Example 2 — Pricing question
**Original**: 「Kindly note our pricing structure as follows:」
**Polished**: 「Pricing's tiered by quantity:」
**Why better**: 直入

## Example 3 ...
```

呢個 file 比抽象規則強 10 倍 —— Claude 參考真實 example 嘅 pattern。

### 4. Install skill 入 Claude

方法視乎你 access level：

**方法 A —— Claude.ai Pro plan**：
- Settings → Skills → Upload custom skill
- Zip 你 skill folder upload
- Claude 自動 parse + 通知 install success

**方法 B —— Claude Code（開發者）**：
- 將 skill folder 放入 `~/.claude/skills/`（user-global）或者 `./.claude/skills/`（project）
- Claude Code 開新 session 時自動 load

**方法 C —— Claude API / SDK**：
- 透過 Agent SDK 嘅 skill parameter 傳入
- 睇 [docs.anthropic.com/skills] 攞最新 API spec

### 5. 測試 skill

返去 Claude 對話，貼一段 draft email：

```
我想 polish 呢段：

「Dear Mr Chan, I would like to inform you that the delivery 
of your order has been delayed due to unforeseen circumstances. 
We sincerely apologize for any inconvenience caused. Please be 
advised that we are working diligently to resolve this matter.」
```

如 skill description 寫得啱，Claude 會：

1. **自動 detect**「呢個係 polish email request」
2. **叫起** skill workflow
3. **應用**你嘅規則 + 參考 example
4. **出** 2-3 個版本 + 建議 1 個

如果 Claude 冇自動叫：

- 你可以明確講「use email-polish-hk skill」
- 或者 review skill description 唔夠具體

### 6. 每週 iterate skill

同 Projects 一樣，第 1 個禮拜 skill 唔完美。每個禮拜五 5 分鐘：

- Review 過去 5 次 invocation
- 揾邊樣你手動改多次 → 加規則
- 揾邊樣 over-rule → 放鬆
- 加新嘅 good example 入參考 file

3 個月後你 skill 自動進化到極準。

## 變化

### 變化 1：多步 skill（workflow chain）

複雜 skill 唔淨止 1 步。例如「由 project notes 出每週報告」：

```markdown
# Weekly report generator

## When to use
User has pasted week's project notes (rough bullet form) and wants summary.

## Step 1: Categorize notes
- Split bullets into: progress / blocker / decision / 跟住做
- 列清楚類別，唔好 mix

## Step 2: Identify themes
- 揾 2-3 個跨類別嘅 recurring theme
- 1 句 summary 每個

## Step 3: Draft report
- 結構：
  - 1 段 executive summary
  - 「This week」section（3-5 個 bullet）
  - 「Blockers」section（最多 3 個）
  - 「Next week」section（3 個 bullet）
- Tone：事實為主，唔好 over-promise

## Step 4: Format
- Markdown
- 唔超過 400 字
- 每個 section 1 個具體數字（具體 > 行貨）

## Rules
- 唔好加「I think」/「probably」—— 你係 report 事實，唔係意見
- 如 notes 不足以分類，反問 user

## Reference
See `reference/past-reports.md` for tone calibration.
```

### 變化 2：Project 同 skill 嘅分別

新手會混淆。快速釐清：

**Project**：對話入面持續 context（例如你公司 brand voice 永遠 active）
**Skill**：特定 workflow 嘅程序（例如「polish email」流程，trigger 時先啟動）

可以同時用：Project 提供長駐 context，Skill 提供任務專屬程序。

例：Bull Kicks Brand project + email-polish skill + sales-pipeline skill + product-description skill = 4 個 skill 喺同一個 project 入面各自可叫。

### 變化 3：將 skill 分享畀同事 / community

你 build 嘅 skill 可以分享。將 skill folder zip + upload 去：

- **Team（內部）**：Claude Team plan 嘅共享 skill library
- **Public（對外）**：GitHub repo + 分享下載連結
- **Anthropic skill marketplace**：部分 skill 可以 submit（check current submission process）

人哋 install 你 skill 後，可以叫 + iterate 自己嘅版本。

如果你 skill 解決一個常見 HK 問題（例如「polish 商業 email」/「draft 道歉信」/「整 IG caption」），分享出去比你一個人用更有價值 —— community 版本會持續改善。

## 一個 mindset

Skill 同 Project 嘅深層價值唔係「自動化」—— 係**強迫將默會嘅 knowledge 講清楚**。

你寫 SKILL.md 嘅過程強迫你講出聲：「我點 polish 一封 email？」「我點揀邊個字好啲？」呢類默會 decision 平時喺你個腦 0.5 秒發生，但講清楚出嚟需要 30 分鐘。

寫完之後你對「自己嘅 craft」嘅理解深一個層次。Claude 變得 well-trained，但真正受益者係你自己嘅思維清晰度。

最後提醒：

- ✅ **第一個 skill 唔需要完美**。Ship 個粗略版本，4 個禮拜後 iterate。
- ⚠️ **Skill 同 system prompt 唔好 overlap 太多**。Project context（長駐）vs skill（workflow）—— 分清楚。
- 🎯 **分享 skill 建立 community 網絡**。HK Claude user community 仲新 —— 早分享 = 早建立聲譽。

Build 第一個 skill 嗰朝你會發現 —— 你以為「我已經喺腦入面做完啦」嘅嘢，講清楚出嚟，竟然有 30 條暗藏規則。Skill = 你 craft 嘅活文件。
