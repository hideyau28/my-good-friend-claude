## 情境

你 admin，公司每星期出 3-5 封 client email。每封流程相同：

1. 你 draft 一段 message
2. 同 Claude 講「polish 呢段 — 要專業但唔生硬，要 HK 客感覺真誠，唔好行貨 form letter 味」
3. Paste 3 個 past good example 做 reference
4. Paste 公司 tone 規範
5. Claude 出 3 個 version
6. 你揀一個 fine-tune

每次 5 段 paste + 揀 = 12-15 分鐘。每星期 5 次 = 75 分鐘。每年 65 小時 paste 同樣嘢。

呢個問題同 Projects 唔同：

- Projects 解決「same context across all chats」
- 但你呢個 use case 係「same workflow across different content」— 每次 polish 唔同 email，但流程一樣

呢個就係 **Claude Skills** 解決嘅問題。

**Skill** = 一個 markdown 文件（加上 optional helper files），描述：
- 點 trigger 個 workflow
- 用咩 reference / rule
- Output 要 follow 咩 structure

Build 完之後 Claude 喺 conversation 入面自動 detect 應該 invoke 邊個 skill，跟住 workflow 行。

呢篇 45 分鐘 build 第一個 skill，往後幾百封 email polish 都唔需要 paste rule。

⚠️ Skills 而家係 Pro / Team / Max plan feature（部分早期版本）。

## 跟住做

### 1. 認識 skill 嘅結構

每個 skill 係一個 folder 入面有：

```
my-skill/
├── SKILL.md              # 主要 instruction file
├── reference/
│   ├── good-examples.md  # 你 past good output (optional)
│   ├── tone-rules.md     # detailed rules (optional)
│   └── templates/
└── scripts/              # 進階：bash / python helpers (optional)
```

最簡單嘅 skill = 1 個 SKILL.md。複雜 skill 加 reference files + scripts。

SKILL.md 嘅 structure：

```markdown
---
name: email-polish-hk
description: Polish English email for HK client — professional but warm, avoid corporate cliche. Trigger keywords: polish, email, client message, 改 email.
---

# Email Polish (HK)

## When to use
- User pastes a draft email + asks for polish / improve / tone-adjust
- 唔 trigger 對於 social media / 內部 message

## Step-by-step
1. 讀 draft 揾 3 樣嘢：unclear sentences / corporate cliches / 過度 hedging
2. 出 2-3 version polished，每個 angle 唔同
3. 結尾標明邊個 version 你 recommend + 1 句 reason

## Rules
- 唔用：「I would like to」/「Please be advised」/「Kindly」
- 改用：「Want to」/「Just wanted to flag」/「Could you」
- 短句 > long sentence
- 結尾唔好 「Thank you for your understanding」 — 改用 「Let me know if anything」

## Reference
See `reference/good-examples.md` for 5 past polished emails.
```

呢個 frontmatter (top YAML block) 嘅 `description` 係**關鍵** — Claude 用呢段判斷「呢個 conversation 應唔應該 invoke skill」。寫具體啲。

### 2. Build 第一個 skill

打開 text editor，create folder `email-polish-hk/`，create file `SKILL.md`，paste 上面 example 改成你 use case 嘅具體內容。

關鍵 fields：

**Name**: kebab-case，唯一識別 skill
**Description**: 包括 trigger keyword + use case 講清楚。e.g., 「Trigger when user asks to polish / 改 / refine an English email for HK client. Skip for casual social posts.」
**When to use** section: 明確界線，等 Claude 知幾時唔好 invoke
**Step-by-step**: 你想 Claude 跟嘅程序
**Rules**: 硬性 constraints
**Reference**: link 去 supporting files

### 3. 加 reference files

`reference/good-examples.md` 例：

```markdown
# Past polished emails (good examples)

## Example 1 — Late delivery notice
**Original**: 「I would like to inform you that the order has been delayed due to unforeseen circumstances.」
**Polished**: 「Just heads up — your order's delayed by 3 days. Quick context: factory had a power issue Tuesday. New ETA Friday. Sorry about this.」
**Why better**: 具體原因 + 具體日期 + 真誠道歉

## Example 2 — Pricing question
**Original**: 「Kindly note our pricing structure as follows:」
**Polished**: 「Pricing's tiered by quantity:」
**Why better**: 直入

## Example 3 ...
```

呢個 file 比抽象 rule 強 10 倍 — Claude reference 真實 example pattern。

### 4. Install skill 入 Claude

方法視乎你 access level:

**Method A — Claude.ai Pro plan**:
- Settings → Skills → Upload custom skill
- Zip 你 skill folder upload
- Claude 自動 parse + 通知 install success

**Method B — Claude Code (developer)**:
- 放 skill folder 入 `~/.claude/skills/` (user-global) 或 `./.claude/skills/` (project)
- Claude Code 自動 load 隨 new session

**Method C — Claude API / SDK**:
- 透過 Agent SDK 嘅 skill parameter 傳入
- See [docs.anthropic.com/skills] for current API spec

### 5. 測試 skill

回到 Claude conversation，paste 一段 draft email：

```
我想 polish 呢段:

「Dear Mr Chan, I would like to inform you that the delivery 
of your order has been delayed due to unforeseen circumstances. 
We sincerely apologize for any inconvenience caused. Please be 
advised that we are working diligently to resolve this matter.」
```

如 skill description 寫得啱，Claude 會：
1. **自動 detect**「呢個係 polish email request」
2. **Invoke** skill workflow
3. **Apply** 你 rules + reference examples
4. **出** 2-3 version + recommend 1

如果 Claude 唔 auto-invoke：
- 你可以明確講「use email-polish-hk skill」
- 或者 review skill description 唔夠具體

### 6. 每週 iterate skill

同 Projects 一樣，第 1 個禮拜 skill 唔完美。每個禮拜五 5 分鐘：

- Review 過去 5 次 invocation
- 揾邊樣你手動改多次 → 加 rule
- 揾邊樣 over-rule → 放鬆
- 加新嘅 good example 入 reference file

3 個月後你 skill 自動進化到極準。

## 變化

### 變化 1：Multi-step skill (workflow chain)

複雜 skill 唔淨止 1 個 step。例如「Generate weekly 報告 from project notes」:

```markdown
# Weekly report generator

## When to use
User has pasted week's project notes (rough bullet form) and wants summary.

## Step 1: Categorize notes
- Split bullets into: progress / blocker / decision / 跟住做
- 列 clear category，唔好 mix

## Step 2: Identify themes
- 揾 2-3 個 recurring theme cross categories
- 1 sentence summary each

## Step 3: Draft report
- Structure:
  - 1-paragraph executive summary
  - 「This week」 section (3-5 bullets)
  - 「Blockers」 section (max 3)
  - 「Next week」 section (3 bullets)
- Tone: factual, 唔好 over-promise

## Step 4: Format
- Markdown
- 唔超過 400 字
- 1 個具體 number per section (具體 > 行貨)

## Rules
- 唔加「I think」/「probably」 — 你 reporting fact, not opinion
- 如 notes 不足以分類, 反問 user

## Reference
See `reference/past-reports.md` for tone calibration.
```

### 變化 2：Project 同 skill 嘅分別

新手會 confused。快速釐清：

**Project**: 同 conversation 入面持續 context (e.g., 你公司 brand voice 永遠 active)
**Skill**: 特定 workflow 嘅程序 (e.g., 「polish email」流程，trigger 時先 activate)

可以同時用：Project 提供 standing context，Skill 提供 task-specific 程序。

例：Bull Kicks Brand project + email-polish skill + sales-pipeline skill + product-description skill = 4 個 skill 喺同一 project 各自 invocable。

### 變化 3：Share skill 畀同事 / community

你 build 嘅 skill 可以 share。將 skill folder zip + upload to:

- **Team (內部)**: Claude Team plan 嘅 shared skill library
- **Public (對外)**: GitHub repo + share download link
- **Anthropic skill marketplace**: 部分 skill 可 submit (check current submission process)

別人 install 你 skill 後，可以 invoke + iterate 自己 version。

如果你 skill 解決一個常見 HK 問題 (e.g., 「polish 商業 email」/「draft 道歉信」/「整 IG caption」)，share 出去比 you alone use 更有價值 — community version 持續改善。

## 一個 mindset

Skill 同 Project 嘅深層價值唔係「自動化」——係 **強迫將默會 knowledge 講清楚**。

你寫 SKILL.md 嘅過程強迫你講出聲：「我點 polish 一封 email？」「我點揀邊個字好啲？」呢類默會 decision 平時 happen 喺你個腦 0.5 秒，但講清楚出嚟需要 30 分鐘。

寫完之後你對「自己嘅 craft」嘅 understanding 深一個層次。Claude 變 well-trained，但真正受益者係你自己嘅 thinking clarity。

最後 reminder：

- ✅ **第一個 skill 唔需要完美**。Ship 粗略 version，4 週後 iterate。
- ⚠️ **Skill 同 system prompt overlap 唔好太多**。Project context (standing) vs skill (workflow) — 分清楚。
- 🎯 **Share skill build community network**。HK Claude user community 仲新 — 早 share = 早建立聲譽。

Build 第一個 skill 嗰朝你會發現 — 你以為「我已經喺腦入面做完啦」嘅嘢，講清楚出嚟，竟然有 30 條暗藏 rule。Skill = 你 craft 嘅活文件。
