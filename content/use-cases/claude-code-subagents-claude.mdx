## 情境

你 senior dev。同事 push 咗一個大 PR — 500 行 change 跨 12 file。

你 review 動作:

1. Read 全部 file diff: 15 分鐘
2. Check test coverage: 5 分鐘
3. Run security scan (manually): 5 分鐘
4. Write thoughtful PR comment: 10 分鐘

Total: 35 分鐘 sequential。

呢個 sequential 嘅根源: **任務之間唔互相 dependent** — code review 唔 wait test coverage 結果，test coverage 唔 wait security scan。但你 default 跑 sequentially because 個腦只能 focus 1 個 task at a time。

**Subagents** 解決呢個: 一個 main session 同時 spawn N 個 subagent，每個 focused 不同 sub-task，並行 run。

**概念**:
- **Main agent** (你)：orchestrate + integrate result
- **Subagents**：parallel workers，each 收 specific task + return result

呢篇拆 subagent setup + 5 個 daily use case scenarios。

⚠️ Subagent 要較高難度 understanding — 適合已用 Claude Code 1-2 個月 + 熟 config files。

## 跟住做

### 1. Subagent 概念清晰啲

```
Main session (you in interactive Claude Code)
  │
  ├── spawn → Subagent 1 (Code reviewer)
  │             └── output: 5 issue spotted
  │
  ├── spawn → Subagent 2 (Test coverage analyst)
  │             └── output: 80% coverage, 3 untested function
  │
  ├── spawn → Subagent 3 (Security scanner)
  │             └── output: no secret leaked, 1 SQL injection risk
  │
  └── integrate → final summary 畀你
```

每個 subagent 獨立 context window，自己嘅 conversation history。Parallelism = 真實 wall-clock 時間慳到。

### 2. Built-in vs custom subagents

Claude Code 自帶一啲 standard subagent type:

- **General-purpose**: open-ended task
- **Explore**: read-only search across codebase
- **Plan**: software architect role planning
- **Code reviewer**: focus on code quality issues

Custom subagents: 你可以 define 自己嘅 (next section)。

### 3. Invoke built-in subagent

喺 Claude Code session 入面，你可以 spawn subagent via Agent tool:

```
Agent(
  description: "Find all auth-related code paths",
  subagent_type: "Explore",
  prompt: "Search the codebase for all files involved in authentication. Look for auth middleware, login routes, session management. Report file paths + brief description of each. Don't read full files, just locate."
)
```

(具體 syntax 視乎 Claude Code version + 你 IDE integration。`claude --help` 顯示 current。)

呢個 spawn 一個 Explore subagent。佢 return 一份 summary，main agent 接住 work on full files。

### 4. Spawn 多個 subagents parallel

關鍵 insight: 你 spawn 多個 subagents 喺 same response，parallel。

Example workflow (PR review):

```
我哋 spawn 4 個 subagents parallel:

Agent({
  description: "Code quality review",
  subagent_type: "general-purpose",
  prompt: "Review the PR diff in current branch. Focus on: error handling, naming, complexity. Output 3-5 specific issues with file:line references."
})

Agent({
  description: "Test coverage audit",
  subagent_type: "general-purpose",
  prompt: "Identify which functions in the PR diff lack test coverage. Check /tests folder. Output: list of untested function names + suggested test scenarios."
})

Agent({
  description: "Security risk scan",
  subagent_type: "general-purpose",
  prompt: "Scan PR diff for security risks: SQL injection, XSS, exposed secrets, unsafe deserialization. Output: risk level + specific code references."
})

Agent({
  description: "Generate PR description",
  subagent_type: "general-purpose",
  prompt: "Read the PR diff and write a 200-word PR description: what changed, why, test plan. Output in markdown."
})
```

4 個 agent 同時 spawn → 並行 work → main session 等 4 個都 done → integrate 4 個 result → 你 see comprehensive review。

Wall-clock time: 6-8 分鐘 vs 30-35 分鐘 sequential。

### 5. Custom subagent definition

Define 可重用 subagent 喺 `~/.claude/agents/` (user-level) 或 `./.claude/agents/` (project-level):

File `code-reviewer.md`:

```markdown
---
name: code-reviewer
description: Specialist for thorough code review. Use proactively after writing significant new code or before commit.
tools: Read, Grep, Bash(git diff:*)
---

You are a senior code reviewer with 15 years experience in JavaScript/TypeScript and Python.

Your job:
1. Read the recent code changes carefully (use `git diff` to see)
2. Identify specific issues in these categories:
   - Bugs / edge case missing
   - Performance concerns
   - Naming / readability
   - Error handling gaps
   - Security issue
3. Output: 5-10 specific findings, each with:
   - File path + line number
   - Issue type
   - 1 sentence explanation
   - Suggested fix

Tone: Direct, specific. No filler. Treat me as peer dev.
```

Save。Claude Code 自動 load 隨 new session。

用:
```
Agent({
  description: "Review my changes",
  subagent_type: "code-reviewer",
  prompt: "Review the last 3 commits in current branch"
})
```

### 6. Daily subagent use cases

✅ **Use case 1: 並行 investigation** (你 read 大 codebase)

```
Agent({ subagent_type: "Explore", prompt: "Find database schema files" })
Agent({ subagent_type: "Explore", prompt: "Find auth middleware" })
Agent({ subagent_type: "Explore", prompt: "Find API route definitions" })
```

3 個並行 investigation，main session 整合結果，你 decide where to focus。

✅ **Use case 2: 多角度 review**

```
Agent({ subagent_type: "security-reviewer", prompt: "Audit auth flow security" })
Agent({ subagent_type: "performance-reviewer", prompt: "Audit auth flow perf" })
Agent({ subagent_type: "readability-reviewer", prompt: "Audit auth flow clarity" })
```

3 個不同角度，同一段 code。Main session 綜合。

✅ **Use case 3: 並行 content generation**

```
Agent({ prompt: "Draft user-facing release notes for these commits" })
Agent({ prompt: "Draft internal team update for these commits" })
Agent({ prompt: "Draft social media announcement for these commits" })
```

3 個 audience，3 種 voice，並行。

✅ **Use case 4: Pre-deploy 並行 checks**

```
Agent({ prompt: "Run npm run test and report failures" })
Agent({ prompt: "Run npm run lint and report errors" })
Agent({ prompt: "Check for hardcoded secrets in diff" })
Agent({ prompt: "Verify all new env vars documented in README" })
```

4 個 deploy gate checks，並行。1 個 fail → halt deploy。

✅ **Use case 5: Bug investigation**

```
Agent({ prompt: "Search for error 'XYZ' in logs" })
Agent({ prompt: "Find similar past issue in GitHub" })
Agent({ prompt: "Check related test cases for coverage gap" })
```

3 個並行調查角度，main agent 整合 findings。

## 變化

### 變化 1：幾時唔好用 subagents

唔適合 parallel:

❌ Task 之間 strict dependency (each depend on previous output)
❌ Single-file edit (no parallelism gain)
❌ 簡單 query (spawn overhead > task time)
❌ Stateful task series (需要 shared context)

✅ 適合:
- Independent investigation
- 多角度 review
- 並行 content gen
- Pre-deploy multi-check
- Multi-source data fetch

Subagent 價值同 task 獨立性成正比。

### 變化 2：Sub-subagents (遞迴)

進階：subagent 自己可 spawn sub-subagents。

E.g.,「Refactor architecture」main agent → spawn「Plan refactor」subagent → 嗰個再 spawn「Read existing auth module」sub-subagent。

3 層 hierarchy 通常夠。深過 = orchestration 複雜度過高。

### 變化 3：Subagent + Skills + Commands 三者組合

Skills tier (per 第 79 篇) 應用「specific procedure logic」。Subagent 應用「parallelism」。

例：
- `/pre-deploy-check` slash command 觸發
- 內 spawn 4 個 subagent (test / lint / security / docs)
- 每個 subagent 內可 invoke skills (e.g., security-audit skill)
- 整合後 single summary report

呢個 3-tier composition (command → subagent → skill) 係 advanced workflow architecture。

## 一個 mindset

Subagent 嘅深層 insight: **CPU / IO parallelism 對 server task 已成熟，但 cognitive task parallelism 對 human 仍然困難。AI subagent 係 the breakthrough**。

之前你個腦 1 thread sequential。即使多 monitor 都唔可以真正 parallel attention。Subagent = 將 parallelism 外判 — 5 個 subagent 同時跑 5 個 task，你 main brain 只負責 orchestration + integration。

呢個 paradigm 顯著改變 senior dev / orchestrator 角色 — 唔再係「solo execution」，而係「multi-agent orchestration」。

最後 reminder：

- ✅ **Subagent 價值大喺「independent task parallelism」**。識別 independent task 係關鍵 skill。
- ⚠️ **Subagent 之間唔 share context**。各自 fresh memory。你 spawn 時必須 include all needed context。
- 🎯 **Custom subagent 1 個月自然會 evolve**。你 daily ship 嗰陣會發現「I keep doing same review pattern」，codify 入 reusable subagent definition。

下個大 PR review 試 4-parallel subagent。30 分鐘 → 8 分鐘。你個腦嘆「點解之前 sequential 咁慢」。
