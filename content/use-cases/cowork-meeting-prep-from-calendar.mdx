## 情境

朝早 9 點，你開電腦，calendar 已經塞滿：

- 9:30 — 同 product manager 傾 Q3 roadmap
- 11:00 — Sales call，個客上次傾過你都唔記得講過咩
- 1:30 — 你個直屬下屬嘅一對一
- 3:00 — 同供應商檢視 proposal
- 4:30 — 跨 team 開 alignment meeting

每個 meeting 你**應該**做嘅 prep：

1. 揾返上次同呢個人傾過咩（搜尋 Gmail / Slack / Notion）
2. 揾返今次相關嘅 doc / spec / 數字
3. 諗 3 條要問嘅問題，至少有個切入點入場
4. 睇下個議程合唔合理

5 個 meeting × 12 分鐘 prep = 一個鐘。但你冇一個鐘——你 9 點開電腦，9:30 已經要入第一個 meeting。所以大部分時間，你係**裸入 meeting**：聽人哋講、唯唯諾諾、行禮如儀，散會先發覺啲嘢應該講而冇講。

呢篇教你設定一次 Cowork，**每朝 8:50am 自動讀你 calendar、對照 email + docs、每個 meeting 出一頁 prep doc 放入 Google Drive**。你 9 點開電腦，5 份 prep doc 已經喺度。

> Caveat：Cowork 要 Pro / Max plan + Claude Desktop。佢會讀你 calendar、emails、Drive 啲嘢，授權之前要諗清楚邊啲帳戶可以授權（建議用工作帳戶，唔好 mix 個人）。**呢個 workflow 全部係唯讀**——出 doc、唔會自動 send email、唔會改 calendar、唔會回覆任何人。

## 跟住做

### 1. 開 Cowork session + 連好帳戶

Claude Desktop（Pro / Max）→ Cowork mode。第一次用要授權以下 access：

- Google Calendar（或 Outlook）— 讀你 events
- Gmail（或 Outlook mail）— 揾返上次 thread
- Google Drive（或 Notion / Dropbox）— 揾相關 doc + 寫低 prep doc

如果你公司用緊 Slack / Teams 又有 MCP connector，順手連埋（變化 2 會用到）。

### 2. 個核心 prompt — 設定一次跑成年

呢個 prompt 重點係教 Claude **點對照**：唔係淨係一口氣攞晒成個 calendar，而係每個 attendee 都要返 Gmail / Drive 揾返 context。

<PromptBlock label="完整 prompt — 每朝自動出 meeting prep">
我想設定一個每朝 08:50 (HKT) 自動跑嘅 workflow。

**觸發：** 每個工作日 (Mon-Fri) 朝早 08:50

**Step 1：讀今日 calendar**

由我嘅 Google Calendar 攞返今日 09:00 之後嘅所有 events。
排除：
- 已取消嘅
- 我拒絕咗嘅
- 純預留時間 / 專注時段（無其他 attendee 嘅）
- 1 小時以內嘅短 sync 但 attendee 超過 8 個嘅（大 town hall，唔需要 prep）

**Step 2：每個 meeting 出一頁 prep doc**

每個保留落嚟嘅 meeting，做：

A. **基本 info**
- Title、時間、地點 / link
- Attendees（標返內部 vs 外部）
- 主辦人

B. **上次 context（最重要）**
搜返過去 90 日 Gmail + Drive：
- 同呢班 attendees（尤其主辦人）最近嘅 email thread — 抽 3 個最相關
- 對話入面提過嘅 decision、commitment、未解決嘅 question
- 任何「我應承過跟進」嘅 item（呢個必出，唔好漏）

C. **相關 docs**
由 Google Drive 揾 3 個最相關嘅 doc / spec / sheet：
- Title + 1 句點解相關
- 直接連結
- 標返 last edited date（太舊嘅要提我留意）

D. **3 條 prep question**
基於 B + C，幫我草擬 3 條我入 meeting 之前要諗清楚嘅問題。
要求：
- 唔好空泛（唔好出「咩係 success metric」呢類阿媽問題）
- 要扣 context（例如「上次你提過 timeline 會 push 1 個月，今次有無 update？」）
- 至少有 1 條係 risk / blocker / 唔肯定嘅地方

E. **Agenda 建議**
如果 calendar event 已經有 agenda：檢查有冇漏 B 入面嗰啲未解決嘅 item。
如果冇 agenda：草擬 1 個 5-bullet agenda。

**Step 3：output 落 Google Drive**

寫入 folder「Daily Meeting Prep / [YYYY-MM-DD]」。

每個 meeting 一個 markdown file：
`[HH:MM] - [meeting title].md`

另外出一個 `00 - Today Overview.md` 列：
- 今日總共幾個 meeting、總時長
- 標 1-2 個「最需要 prep」嘅（即 stakes 高、attendee 高層、或者上次有 unresolved item）
- 標 1-2 個「可以 deprioritize」嘅（純通報、唔需要你發言）

**Step 4：通知我**

寫好之後寄一封短 email 畀我，主旨「今日 Meeting Prep Ready — X 個 meeting」。
內容：folder link + 「最需要 prep」嗰 1-2 個 meeting 嘅 title + 直接 link。
唔好喺 email 入面 dump prep 內容——只係 pointer。

**重要限制：**
- 全程唯讀：唔好覆任何 email、唔好改 calendar、唔好 share 任何 doc 出去
- 如果搜唔到上次 context（例如新 attendee），明講「冇 prior context」，唔好作
- 唔好交叉比對任何標咗「private / personal」嘅 calendar 或 email
- 機密嘅 thread（標咗 機密 / HR / legal）只列存在，唔好抽內容

設定好之後同我核對一次，行一次預演睇 output，先至 schedule。
</PromptBlock>

### 3. Output 格式 + Dry run

第一次跑**唔好直接 schedule**。叫 Cowork：「即時跑一次今朝 calendar，show 我 5 份 prep doc，唔好真係寫入 Drive」。

睇住佢出嘅 output，檢查幾樣嘢：

- **Context 抽得準唔準**——5 份入面有冇 1 份係抽錯 thread / 攞錯 doc
- **3 條 question 嘅質素**——有冇真係扣到 context，定係籠統
- **Agenda 建議**——合唔合理，有冇漏明顯嘅 item
- **隱私邊界**——有冇唔小心抽咗 私人 email / confidential thread

執完先啟用 schedule。輸出落 Google Drive 之後，你可以將 prep doc folder 置頂落 desktop 或者收藏，每朝一 click 就到。

### 4. 首週微調

頭一個禮拜出嘅 5 × 5 = 25 份 prep doc，你會發現規律：

- **某啲 attendee Cowork 唔識搵 context**（例如名字喺 Gmail 用英文、calendar 用中文）→ 加一個別名對照落 prompt
- **3 條問題太籠統** → 加例子落 prompt，show 你想要嘅問題風格
- **Agenda 同主辦人原本嘅議程撞**→ 改成「淨係喺主辦人冇定 agenda 時先草擬」
- **某啲定期短 meeting（daily standup）唔需要 prep**→ 加排除規則

呢個 iteration 大約 2 個禮拜會穩定。穩定之後幾乎唔需要再執——除非你跳 team / 換 role。

## 變化例子

### 變化 1：唔用 Cowork，每朝手動觸發

Pro 用家冇 scheduled task，但用 Claude Desktop + MCP 一樣做到，淨係要你每朝自己觸發。

<PromptBlock label="變化 1 — 手動觸發版">
我每朝 8:45 開電腦會手動觸發你一次。

每次跑：
- 讀今日 Google Calendar
- 對每個 meeting，做上面 Step 2 嘅 A-E
- output 唔需要寫 Drive，直接喺呢個 chat 出 markdown 我自己複製
- 結尾列出「最需要 prep」嗰 1-2 個

我會儲存個 prompt 做 [快速命令]，每朝撳一次。
</PromptBlock>

呢個 setup 嘅取捨：你要記得觸發（一兩個禮拜未養成習慣會漏），但你可以即時跟進、即時微調。

### 變化 2：加 Slack / Teams DM context

如果你公司 day-to-day 喺 Slack 傾，淨係 email 抽 context 唔夠完整。

<PromptBlock label="變化 2 — 加 Slack DM context">
喺 Step 2 嘅 B「上次 context」加一層：

由 Slack 搜過去 30 日：
- 同每個 attendee 嘅 DM
- attendees 都 in 嘅 channel 入面，最近 提及過 meeting 主題 keyword 嘅 message

抽 3 個最相關，列：
- Channel 或 DM
- 日期
- 1 句撮要

呢層尤其捉到啲「Slack 講咗、email 冇跟進」嘅 commitment——呢啲最易遺忘。

私隱限制同 email 一樣：private channel / 標 機密嘅 thread 只列存在，唔抽內容。
</PromptBlock>

### 變化 3：Founders / managers 版 — 1:1 meeting prep

你管緊團隊，每星期 5-8 個一對一。每個一對一嘅 prep 同普通 meeting 唔同：重點係**上次 action item follow-up + 對方士氣**。

<PromptBlock label="變化 3 — 1:1 meeting 專屬 prep">
如果今日 meeting attendee 得 1 個內部 direct report，當佢做 1:1 處理：

Step 2 嘅 B「上次 context」改成：
- 揾返上次同呢個人嘅 1:1 prep doc（如果有）
- 抽出上次嘅 action item：佢應承過嘅 + 我應承過嘅
- 標返：邊啲已完結、邊啲仲處理緊、邊啲過期

Step 2 嘅 D「3 條 prep question」改成：
1. 一條 follow-up：上次 action item 嘅進度
2. 一條 前瞻性：佢手頭 project 嘅 blocker / risk
3. 一條 personal：基於最近 Slack / email 嘅 signal（例如最近 PR review 嘅 turnaround、休假 pattern、佢主動提過嘅長遠 goal）— 唔好太生硬，搵唔到信號就跳過

額外加 Step 2.5：
- 列我上次 1:1 應承過佢但未做嘅嘢（呢個係**最容易失信**嘅地方，必出）
- 列佢最近 3 個 PR / project deliverable 嘅 summary，畀我有具體嘢可以讚

呢類 prep 對保住團隊 trust 嘅 ROI，遠高過任何 sales meeting prep。
</PromptBlock>

## 拆解：點解會 work，同點解會爆

跟到上面就已經用得。下面呢段係畀**想由「試咗一朝好靚仔」做到「跑足半年都信得過」**嘅人——初學者可以跳過，唔影響你跟住做。

自動化最唔老實嘅地方係：**第一朝 prep doc 出晒、靚到你心動，唔代表第二十朝仲係咁。** 呢個 workflow 實際會喺以下幾個位爆，你要預咗：

**1. Attendee 名字對唔到（最高頻爆）**
Calendar event 入面 attendee 係「wklee@company.com」，但 Gmail thread 嘅寄件人係「William Lee」、Slack 個 display name 係「阿威」。Cowork 搜返嘅時候未必識 join 三個身份。
- 會爆成點：搵唔到 prior context，prep doc 寫「冇 prior context」——但其實你哋傾過好多次，只係名字冇 match 到。
- 點救：prompt 入面加一個「Attendee alias」字典，例如「wklee = William Lee = 阿威」。頭一兩週邊個搵唔到就即補。

**2. Scheduled task 靜默死去（最危）**
Cowork scheduled job token 過期、API 連接斷、Gmail OAuth revoke 之後，job 可能靜默跳過、唔寫 Drive、又唔寄通知 email——你以為 prep 喺度，其實根本冇跑。
- 會爆成點：你 9 點開電腦揾唔到 prep doc，已經係 9:15，第一個 meeting 15 分鐘後。最壞情況係你唔留意 folder，裸入 meeting 渾然不覺。
- 點救：prompt 內「如果冇法寫入 Drive，至少寄一封錯誤通知 email 畀我」；另外每隔一段時間手動核對 Drive folder 有冇新增。發現斷咗及早 reconnect。

**3. 機密 thread 意外寫落 Drive**
你授權咗 Gmail + Drive，但 HR offer letter、個人醫療假申請、薪酬談判 thread 都係喺同一個 Gmail 帳戶。Prompt 寫咗「標咗機密只列存在」，但「機密」係你主觀 label，唔係 Gmail 官方 tag——Claude 冇辦法可靠識別所有機密郵件。
- 會爆成點：敏感內容被抽進 prep doc、寫落 shared Drive folder，同一 folder 有 edit 權限嘅同事就可以睇到。
- 點救：建立一個「DO NOT READ」Gmail label，所有機密郵件手動 tag；prompt 加「任何 label 含 'do-not-read' 嘅 thread 完全跳過」。比喺 prompt 講「機密嘢唔好抽」更可靠。

**4. Meeting 多嗰日 context 被截斷**
今日 8 個 meeting，每個都要搜 90 日 Gmail + Drive，context window 一次過吞唔落咁多資料，系統靜默截斷，某幾份 prep doc 嘅 B 部分出嚟係空或者半殘——但唔會有 error message 提你。
- 會爆成點：你以為 8 份 prep doc 都齊，但其中 2-3 份嘅「上次 context」部分係白板。你唔睇唔知。
- 點救：prompt 加一句「如果某份 prep doc 搵唔到 context 係因為資料太多，明講『context 資料量超限、請手動補充』，唔好靜默留空」。另外可以將 meeting 分兩 batch 跑（例如上午、下午各一次）。

**5. Claude「補充」咗假 commitment**
搵唔到相關 email / doc，但 Claude 覺得唔好出「冇嘢」，就根據 meeting title + attendee 推斷你「可能」應承過啲咩、「可能」有咩 decision。呢啲係幻構，唔係真實記錄。
- 會爆成點：你帶住一條「上次你答應過 X」嘅假問題入 meeting，被對方當場打臉，trust 即時爆。
- 點救：prompt 已有「搵唔到就明講冇 prior context，唔好作」——**但你 dry run 嗰陣要特別測試呢個邊界**：故意揀一個從未 email 過嘅新 attendee，確認 Claude 真係出「冇 prior context」而唔係生出嘢嚟。定期抽查，唔好以為設定一次就永遠 ok。

呢幾個位，就係「第一朝靚仔」同「跑足半年仍然信得過」之間嘅距離。

> Meeting prep 嘅問題從來唔係「冇時間 prep」——係**你最精神嗰 30 分鐘（朝早 9 點）用嚟做機械式 search**，所以入到 meeting 已經攰。Cowork 嘅槓桿：將機械式 search 推前到 8:50am 之前，你 9 點開電腦見到嘅係 **判斷題**，唔係 **搜尋題**。
