## 情境

你裝咗 Claude Code 兩個月，每日用，工作流順到唔捨得放低。

第一個月 bill 出嚟：US$340。

你嚇親。攤開 usage dashboard 望真啲：

- 70% token 用咗喺「rename variable」、「move file」、「regenerate test fixture」呢類 grunt work
- 20% 用喺日常 feature dev
- 10% 用喺真係要諗嘢嘅 architecture / 拆 bug

而呢三類 task，全部用緊同一個 model — Sonnet 4.5。

呢個預設邏輯有問題：

- **Sonnet 強，但 rename 一個 variable 用佢係燒銀紙**。Haiku 4.5 做呢樣嘢有 90% 質素，價錢得 1/4
- **真係要諗嘢嘅 task 反而用緊「平衡型」**。複雜 race condition、tricky refactor，你需要嘅係 Opus
- **Claude Code 識揀 model，但你冇教過佢點揀**。佢只能用預設

呢篇教你建立 task-to-model routing。3 個 slash command、1 個 CLAUDE.md routing rule、加 subagent dispatch — 跟住做，下個月 bill 落 50%-70%，質素冇跌反而升（因為複雜 task 開始用 Opus）。

## 跟住做

### 1. Model price comparison + use case mapping（5 分鐘）

先建立心智 model。3 隻 model 嘅 input/output 價錢（per 1M tokens，2026 May 報價）：

| Model | Input | Output | 適合 |
|---|---|---|---|
| Haiku 4.5 | US$1 | US$5 | Grunt work、平行 task、boilerplate |
| Sonnet 4.5 | US$3 | US$15 | 日常 feature dev、code review |
| Opus 4.7 | US$15 | US$75 | Architecture、複雜 debug、長 reasoning |

Opus output 係 Haiku 嘅 15 倍。但你日常根本唔需要 Opus 級數推理。

Task-to-model 對照：

- **Haiku**：rename / move file、generate test data、format code、寫 commit message、簡單 regex、search 用 grep 點寫
- **Sonnet**：寫新 component、debug logic bug、refactor 一個 module、code review
- **Opus**：揀 architecture（monolith vs microservice）、race condition、performance bottleneck、規劃多 module 改動

### 2. /haiku /sonnet /opus slash command setup（10 分鐘）

喺 project root 整 `.claude/commands/` folder：

```bash
mkdir -p .claude/commands
```

整三個 slash command 檔。`.claude/commands/haiku.md`：

```markdown
---
description: Switch to Haiku 4.5 for grunt work
---

切去 Haiku 4.5。呢個 model 適合：rename、move、boilerplate、簡單 regex、generate test fixtures。

執行：`/model claude-haiku-4-5`

之後請用最直接方式完成 task，唔好過度解釋。
```

`.claude/commands/sonnet.md`：

```markdown
---
description: Switch to Sonnet 4.5 for feature dev
---

切去 Sonnet 4.5。呢個係日常 default：寫 component、debug、refactor 一個 module。

執行：`/model claude-sonnet-4-5`
```

`.claude/commands/opus.md`：

```markdown
---
description: Switch to Opus 4.7 for architecture / deep reasoning
---

切去 Opus 4.7。只用喺：architecture 決定、race condition、performance bottleneck、跨 module 規劃。

執行：`/model claude-opus-4-7`

之後用 extended thinking，思考多角度。
```

之後喺 Claude Code session 打 `/haiku`、`/sonnet`、`/opus` 就秒切。

### 3. CLAUDE.md routing rule + subagent dispatch（10 分鐘）

喺 project root `CLAUDE.md` 加一節：

```markdown
## Model routing

Default：Sonnet 4.5。

升 / 降 model 嘅準則：
- Task 屬於 rename / move / boilerplate / format / 簡單 regex → 主動建議切 Haiku
- Task 涉及 architecture、race condition、跨 3+ files 嘅規劃 → 主動建議切 Opus
- 唔肯定 → 留喺 Sonnet

Subagent dispatch：
- 用 Task tool 分派 grunt sub-task 嗰陣，明確 spec `model: claude-haiku-4-5`
- 例如「rename 30 個 file 入面個 helper 名」、「regenerate 50 個 test fixture」 → 用 Haiku subagent 平行跑
```

之後你問 Claude「rename 哂呢個 service 入面所有 oldName」，Claude 會自己提：「呢個 grunt work，建議 `/haiku` 切先做。」

Subagent 平行跑嘅例子（喺 prompt 入面寫）：

<PromptBlock>
分派 5 個 Haiku subagent 平行做以下嘢：
1. Rename `oldHelper` 做 `newHelper` 喺 [src/services]
2. Update import path 喺 [src/components]
3. Regenerate test fixture 喺 [tests/fixtures]
4. Update doc reference 喺 [docs/]
5. Update CHANGELOG entry

每個 subagent 用 `model: claude-haiku-4-5`，跑完匯總。
</PromptBlock>

Haiku 平行 5 條，總成本可能仲低過用 Sonnet 順序做一條。

### 4. Monthly cost audit ritual（5 分鐘）

每月 1 號做：

1. 開 [console.anthropic.com](https://console.anthropic.com) → Usage
2. Filter「by model」睇分佈
3. 健康 ratio：Haiku 50-60% / Sonnet 30-40% / Opus 5-10%
4. 如果 Sonnet > 70% → 你 grunt work 冇切 Haiku
5. 如果 Opus > 20% → 你濫用 Opus，部分 task Sonnet 已經做到

調整完，下個月再睇。3 個月之內，你會 calibrate 到自己工作模式嘅最佳分佈。

## 變化

### 變化 1：Solo dev 緊 budget 版

每月想壓喺 US$50 以下。預設反過嚟 — 預設 Haiku，escalate 先用 Sonnet。

`CLAUDE.md`：

```markdown
## Model routing (budget mode)

Default：Haiku 4.5。
只有以下情況 escalate Sonnet：
- 寫新 feature（>50 行 new code）
- Debug 一個你形容係「tricky」嘅 bug
- 我明確打 /sonnet

絕對唔好自動切 Opus。我要用就會打 /opus。
```

呢個模式：質素稍跌、月費可以喺 US$30-50 之間。Side project / 學習階段啱用。

### 變化 2：Agency 多 client 版

接 5-10 個 client，要 allocate cost。每個 client 一個 folder + 一個 CLAUDE.md。

每個 folder 嘅 `.env.local`：

```bash
ANTHROPIC_API_KEY=sk-ant-...
ANTHROPIC_PROJECT_TAG=client-acme
```

Anthropic console 入面開「Projects」，每個 client 一個 project，分開 billing。月底直接出 invoice line item。

Routing rule 按 client 嘅 budget 調 — premium client 容許 Opus，cost-sensitive client 預設 Haiku。

### 變化 3：Education / learning 版

你學緊新 framework / 新 concept，唔係趕住交嘢。

策略反過嚟：

- **Opus for explain**：你唔明嘅 concept、想要深入解釋 → 直接 `/opus`。Opus 嘅解釋有層次、舉例貼題
- **Sonnet for hands-on**：跟住寫 code、做練習 → Sonnet
- **Haiku for repetitive drill**：generate 10 個變化 example、寫 flashcard → Haiku

學習階段「貴 model 用喺 explain」其實划算 — 你慳嘅係「Google 唔到答案嗰兩個鐘」。

## 一個心態

> 預設係給唔識揀嘅人用嘅。識揀嘅人，每個 task 都揀啱 tool。

Claude Code 預設 Sonnet 唔係佢嘅錯 — 係佢未識你嘅工作模式。你 install 第一日，佢冇辦法分得清「rename」同「architecture」對你嘅難度差距。

但兩個月之後，你應該識自己。應該知道：

- 自己 70% 時間做緊咩
- 邊類 task 真係需要推理
- 邊類 task 只係要「快、準、唔錯」

Model routing 嘅意義唔係慳錢 — 慳錢只係 side effect。真正意義係：**你開始 treat AI 做工具，唔再 treat 做萬能 oracle**。Haiku 處理 grunt、Sonnet 出活、Opus 做 deep work — 你嘅腦袋都係咁分工。

最後提醒：

- ✅ **第 1 個月實驗，第 2 個月固定 ratio**。唔好一裝 routing rule 就期待即刻慳 70%
- ⚠️ **唔好過度 micro-optimize**。為咗慳 US$5 打 30 次 `/haiku` 切過嚟切過去，唔抵
- 🎯 **`/cost` 喺 session 入面睇即時 token 消耗**。一個 task 跑完 check 一次，幾日就 calibrate 到

下個鐘整 3 個 slash command + 改 CLAUDE.md。月底先睇 bill 對比。
