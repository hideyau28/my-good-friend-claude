## 情境

你新入職 / 接手一個 project。 開 repo：

- **5 年舊 codebase**
- **150 files、20K+ lines**
- **無 README**（或 only 「How to set up」 嘅 outdated draft）
- **無 architecture doc**
- **3 個原作者已 left company**
- **CI 跑得過，但無人解釋 dep 之間嘅 relationship**

你第一週要 「make sense of」 呢個 codebase 先 ship 任何 feature。 Senior dev 嘅 default workflow：

1. Open random files、scroll、try-to-pattern-match（dazed）
2. Run `grep` 揾類似 keyword
3. 問 ex-colleague 喺 Slack（response 慢 / 已 离职）
4. 1 個禮拜後仲係 confused

呢篇用 Claude Code **自動 map 個 codebase**，2 個鐘出一份 onboarding doc，比起 1 週嘅 stumble-around 快好多。

## 跟住做

### 1. Codebase initial scan

```bash
cd legacy-project
claude
```

入 Claude Code 之後：

<PromptBlock label="第一步 — Codebase architectural mapping">
我啱啱 inherit 呢個 codebase。冇 README 同 architecture doc。我要 1-2 hour 內 understand 個 project。

請：

1. **Scan repo structure**：top-level folder + 主要 sub-folder，2-3 sentence 描述每個負責乜
2. **Identify entry points**：main、index、server start file、test runner
3. **Map main data flow**：由 entry 開始，揾 1-2 個 most important code path（user request → business logic → response）
4. **Tech stack inventory**：language、framework、major library、build tool、test framework
5. **Project age signals**：package.json / pyproject.toml 嘅 dependency 是否 outdated、是否有 lock file inconsistency、CI 文件嘅 latest update

Output 一份 「Onboarding doc v0」（markdown），約 1-2 頁。 寫成新 dev 嘅 day 1 read。

唔好 hallucinate——任何唔肯定嘅嘢標 「⚠️ Needs verification」。
</PromptBlock>

Claude Code 會花 5-10 分鐘 scan，出 initial doc。

### 2. Identify hotspots

<PromptBlock label="第二步 — 揾出 hotspots">
基於 Part 1 嘅 mapping，幫我揾 5 個 「hotspot」——即係：

- **最複雜 file**（最多 line / 最多 function / 最 nested logic）
- **最 frequently changed file**（用 git log 揾 recent 1 年 commit 最多嘅）
- **最 critical 嘅 business logic**（如 payment / auth / data layer）
- **最 fragile 嘅 file**（無 test cover、或 test cover 但 fail rate 高）
- **最 「smells legacy」 嘅 file**（commented-out code 大量、TODO 5+ 個、known anti-pattern）

每個 hotspot：file path、reason、estimated complexity（1-5 star）、recommended first action（read / refactor / test / leave alone）。
</PromptBlock>

### 3. 揾 dead code

<PromptBlock label="第三步 — Dead code detection">
請幫我 identify potential dead code：

- Exported function 從未被 referenced
- Branch / case 入面 unreachable code
- Commented-out code 超過 1 年未 touch
- Environment variable 喺 .env.example 但 codebase 從未讀

Output 一份 「dead code candidates」 list。 每個附：
- File + line
- Confidence（high / medium / low）
- 建議 next action：delete / migrate to deprecated folder / 留 + add TODO

Caveat：呢類 detection 永遠有 false positive（reflection-based reference、external API contract 等）。 行動之前 manual verify 每個 high-confidence candidate。
</PromptBlock>

### 4. Generate onboarding doc 落 repo

<PromptBlock label="第四步 — Generate final onboarding doc">
基於 Part 1-3 嘅 findings，整一份完整嘅 onboarding doc 寫落 `docs/ONBOARDING.md`：

Structure：
1. **Project at a glance**（1 段 + tech stack table）
2. **Architecture diagram**（用 ASCII / Mermaid）
3. **Setup instructions**（refer 現有 setup script，加 caveat 我發現嘅 known issue）
4. **Main code paths**（2-3 個 walk-through）
5. **Hotspots to know**（point new dev to critical files）
6. **Known tech debt**（list dead code、deprecated pattern、需要 refactor 嘅地方）
7. **Onboarding checklist**（new dev day 1-7 要做嘅嘢）

寫成 new senior dev day 1 嘅 reference。 Tone：直接、honest about complexity（唔扮 「easy onboarding」）。
</PromptBlock>

### 5. Iterate as you learn

Onboarding doc v0 唔係 perfect。 接住 2-4 週你做 feature work 嘅時候，**每次 reveal 新 understanding 就 update doc**。 

呢個 living document 嘅 ROI：3 個月後下個 new dev 入嚟，佢有你嘅 documented learning，唔需要由零開始。

## 變化例子

### 變化 1：Specific debugging exploration

<PromptBlock label="變化 1 — 揾 specific bug 嘅 root cause">
我撞到一個 bug：[描述 symptom]。

請幫我：
1. 由 symptom 倒推 likely root cause（list 3-5 個 hypothesis 按 likelihood 排）
2. 對每個 hypothesis，指引我 verify 嘅 file / function
3. 由 most likely 開始 walk through code

呢個係「fishing expedition」唔係 mechanical fix——你係我嘅 senior dev pair。
</PromptBlock>

### 變化 2：Test coverage audit

<PromptBlock label="變化 2 — Test gap analysis">
請 audit 我 test coverage：

1. List test framework + 而家 coverage %
2. Identify 5 個 critical business logic file but 缺 test
3. 對每個建議 minimum test case（happy path + 2 edge cases）
4. Recommend test priority：邊個寫先（risk-weighted）
5. Estimate effort for catch-up to 60% coverage（baseline target）
</PromptBlock>

### 變化 3：Modernization roadmap

<PromptBlock label="變化 3 — Tech upgrade roadmap">
我想 modernize 個 codebase（upgrade dependency、refactor legacy pattern）。

請：
1. Inventory deps + 標 outdated（major version behind）
2. Identify breaking changes（upgrade 邊個 dep 會 cascade）
3. Suggest sequencing：邊個 upgrade 先做（low-risk）、邊個 wait（high-risk）
4. Estimate effort + risk
5. 建議 「strangler pattern」 嘅 refactor approach（incremental，唔 big-bang）

呢個係 3-6 個月 roadmap，唔係 1 週做完。
</PromptBlock>

## 一個 mental discipline

接手 legacy codebase 嘅最大 trap：**「呢個 code 寫得好差，我要 全部 rewrite」**。

幾乎永遠錯。 原因：

- 你睇唔到 original 寫者面對嘅 constraint（time、tool、team context）
- Existing code 有 5 年嘅 production-tested bug fix you don't see
- Rewrite 嘅 cost 永遠超 estimate 2-3 倍

正確 mindset：**respect inherited code as if it's earnest first attempt; 用 understanding 之後嘅 strategic improvement, 唔好 ego-driven rewrite**。

Claude Code 嘅 leverage 喺呢個 understanding stage——將 「我要 1 個月 read code」 嘅 friction，降到 「2 個鐘 mapping + 持續 update」。

呢個 unlock 嘅唔係 「rewrite」，係**有 informed perspective 嘅 incremental improvement**。 senior dev 嘅真實 value，係 know what to keep + what to change。 Claude Code 幫你 reach 嗰個 informed state，剩低個 judgment 仍然係 you。
