## 情境

你係 SaaS 或者電商創辦人，盯住 5 個直接對手：

- 佢哋幾時加價、減價
- 邊個出咗新 pricing tier
- Landing page 嘅 hero copy 點變（反映佢哋市場定位點轉）
- 邊個出咗新 feature、邊個靜雞雞 sunset 咗一個 plan

呢類資訊唔係競爭機密，係佢哋自己擺喺公開網頁。問題係你要每週親手開 5 個 tab、㩒上㩒落、憑記憶對比上週見到嘅版本——通常做兩個月就會放棄。

呢樣雜務嘅特性啱晒 Cowork：重覆、結構化、輸出明確（一頁 brief）、容錯率高（出草稿你睇）。每週一朝 7am 跑一次，你 9am 返到公司就有杯咖啡同一頁 competitive brief 等你。

> 利益申報：呢篇講嘅係讀**公開**嘅 landing page + pricing page。唔好用呢個流程去爬需要登入嘅 dashboard、付費內容、或者你冇權限睇嘅 staging URL。技術上 Claude 唔會幫你做呢啲，但都要講清楚：呢條線過唔得。

## 跟住做

### 1. 整理對手清單 + 做一次基線 snapshot

第一步係手動嘅，做一次就得。開一個 markdown file：

```
~/Documents/competitive-monitor/
├── competitors.md         ← 你要 track 嘅 URL list
├── snapshots/
│   ├── 2026-W21/          ← 每週嘅 fetch 結果
│   └── archive/           ← 舊週嘅
└── briefs/                ← Cowork 出嘅週報
```

`competitors.md` 應該係：

```
## Competitor A
- Landing: https://competitor-a.com
- Pricing: https://competitor-a.com/pricing
- Changelog: https://competitor-a.com/changelog
- Positioning: 「all-in-one for SMB」
- 上次 major change: 2026-04 加咗 enterprise tier

## Competitor B
...
```

「Positioning」同「上次 major change」呢兩個欄好重要——畀 Cowork 一個基線，知道乜叫「值得標出嚟」、乜叫「行貨小改」。

### 2. 核心 diff prompt

<PromptBlock label="完整 prompt — 每週對手監察">
我要設定一個每週一朝 7am 自動跑嘅 competitor monitoring 任務。

**輸入：**
- `~/Documents/competitive-monitor/competitors.md`（5 個競爭對手嘅 URL + 我嘅背景筆記）
- `~/Documents/competitive-monitor/snapshots/` 入面上週抓取嘅結果（用嚟對比）

**每週一 7am 跑：**

第一步：抓取每個對手嘅 landing page + pricing page + changelog（如果有）
- 只讀公開 URL
- 抽出純文字內容（去走 nav / footer / cookie banner 等雜訊）
- 存落 `snapshots/2026-WXX/competitor-a-landing.txt` 等

第二步：對比上週版本
- 同 `snapshots/2026-W(XX-1)/` 入面對應檔案做 diff
- 識別三類變化：
  1. **Pricing change** — 任何數字、tier name、feature gating 嘅改動
  2. **Positioning change** — hero headline、subheadline、main CTA 嘅改動
  3. **Feature announcement** — 新 feature copy、新頁、changelog 入面嘅 entry

第三步：每個變化寫 3-5 句 brief
格式：
- **[Competitor] — [變化類別]**
- 變化內容（before → after）
- 我嘅判斷（呢個對我哋定位嘅意義，1-2 句）
- 建議跟進（要唔要應對 / 觀察就得 / 唔關事）

第四步：出一頁 weekly brief
- 檔案路徑：`briefs/2026-WXX-competitive-brief.md`
- 最頂 TL;DR：3 句講晒呢週最值得知嘅
- 跟住每個對手一個 section
- 最後 section：「冇變化」嘅對手都要列（等我核對冇漏咗）

第五步：email 我
- Subject：「Week XX competitive brief — [N] changes detected」
- Body：上面 TL;DR + brief 嘅檔案路徑
- 唔好成個 brief 都貼晒入郵件，等我自己開檔案睇

**重要限制：**
- 只抓取我清單入面嘅 URL，唔好自己跟住 link 去其他頁
- 唔好嘗試登入任何要授權先睇到嘅內容
- 如果某個 URL 收 403 / 404，標示出嚟，唔好猜內容
- Pricing 嘅變化要引用原文（照抄數字 + tier name），唔好自己改寫
</PromptBlock>

### 3. Weekly brief 格式 — 你 9am 睇 5 分鐘

第一次跑完之後檢查吓輸出嘅排版。理想嘅 brief 應該係：

```
# Week 21 Competitive Brief (2026-05-25 → 2026-05-31)

## TL;DR
- Competitor A 加咗 $20/月喺 Pro tier，feature 無增加
- Competitor C 改咗 hero copy，由「for teams」轉「for enterprises」
- 其餘 3 個 competitor 無變化

## Competitor A — Pricing change
Pro tier：$49 → $69/月。Feature list 完全冇變。
我嘅 take：純粹加價測試，可能反映佢哋 retention 唔錯所以敢試。
建議：觀察兩週睇有冇回滾。我哋 Pro tier 暫時唔需要跟。

## Competitor C — Positioning change
Hero：「Project management for teams」→「Enterprise-grade project management」
Sub：CTA 由「Start free」變「Talk to sales」
我嘅 take：明顯上移 ICP，可能放棄 SMB 市場。
建議：呢個係我哋 SMB 嘅機會窗口，下季 marketing 可以揀呢個角度打。

...
```

3-5 句一個 entry。多咗你唔會睇，少咗冇 context。

### 4. 月度策略檢視 — 用 4 週 brief 做趨勢分析

每週嘅 brief 各自獨立冇咩用——真正嘅價值喺月底睇到嘅趨勢。

每個月最後一個禮拜五跑多一個任務：

<PromptBlock label="月度趨勢整合">
讀取 `~/Documents/competitive-monitor/briefs/` 入面過去 4 週嘅 brief。

整一份 monthly strategic memo：

1. **市場趨勢**：5 個對手加埋睇，呢個月有冇集體郁動？（例：3 個都加價 → 行業嘅付費意願上升）
2. **個別對手嘅戰略軌跡**：每個對手過去 4 週嘅變化串起嚟，反映佢哋 roadmap 嘅方向
3. **空白市場**：5 個對手都冇做嘅 segment / feature / positioning，可能係我哋嘅機會
4. **威脅信號**：邊個對手嘅變化最值得我哋警戒，點解

寫畀我自己睇——直接、有立場、唔好「呢個就要睇情況」嗰種模稜兩可。
</PromptBlock>

呢一份 memo 一個月做一次，係你季度規劃會議嘅重要參考。

## 變化例子

### 變化 1：加埋廣告監察

<PromptBlock label="變化 1 — 加埋 Meta ads library + Google ads transparency">
喺原本流程上面加多一步：

第一步加：抓取 Meta Ad Library 入面每個對手嘅 active ads（公開資料）
- 用 URL：https://www.facebook.com/ads/library/?q=[competitor name]
- 抽出 ad creative 嘅文字、started running 日期
- 留意：同上週對比，邊啲 ad 新出、邊啲落咗、邊啲改咗 copy

第二步加：Google Ads Transparency Center 做同樣嘢
- URL：https://adstransparency.google.com/?domain=[competitor.com]

加入 brief 嘅新 section：「廣告動向」
- Active ads 嘅 copy 主題（反映緊乜 messaging）
- 新出嘅 ad 反映乜 campaign focus
- 落咗嘅 ad 反映邊種 messaging 唔 work

呢個對靠 paid acquisition 嘅 startup 特別有用——你睇到對手嘅 paid messaging 變化，往往比 landing page 更快反映佢哋定位上嘅試探。
</PromptBlock>

### 變化 2：加埋新聞 + 社交討論

<PromptBlock label="變化 2 — 加新聞 + 社交監察">
我想擴展呢個監察去埋：

- Google News：每個對手名搜過去 7 日（用 site:news.google.com 嘅 query format）
- Reddit：subreddit 入面有冇人提到對手（例：r/SaaS、r/Entrepreneur、行業特定 sub）
- LinkedIn：對手公司 page 嘅新 post（公開內容）

加入 brief 嘅新 section：「外部聲音」
- 媒體覆蓋（PR placement、融資公告、人事變動）
- 用戶討論：邊啲係正面、邊啲係負面、有冇重複出現嘅投訴
- 我嘅判斷：呢啲外部信號同 landing page 變化加埋睇，反映佢哋對外講法夾唔夾得埋

注意：唔好嘗試讀 private LinkedIn post、唔好爬需要登入嘅論壇。
</PromptBlock>

### 變化 3：自家 product team 內部用

<PromptBlock label="變化 3 — 跟自己嘅 release notes">
我哋自己係 SaaS，每個 sprint 出 release notes。

請改造呢個流程嚟做內部用：
- 監察我哋自己嘅 changelog page + 過去 4 週嘅 release notes
- 同我哋自己嘅 positioning doc（內部 PRD / messaging guide）對比
- 標示出：邊啲我哋出咗嘅 feature 喺 marketing site 上面冇反映、邊啲 messaging 過時、邊啲 pricing tier 嘅 feature list 同實際 product 對唔上

每週一畀我哋一份 product / marketing alignment brief：
- 「呢個 feature 上週出咗，但 marketing page 入面冇提」
- 「Pricing page 寫嘅 Pro tier 包 X，但 X 已經 sunset 咗」
- 「我哋自己嘅 hero copy 同上月 positioning workshop 嘅結論已經偏咗」

呢個對 10-30 人嘅 startup 特別有用——marketing / product / sales 之間嘅步調通常喺呢個規模開始偏離，定期 audit 比起每季打仗高效得多。
</PromptBlock>

## 點解競爭監察啱 Cowork 做

> 競爭監察呢樣嘢嘅本質係：每次都係低風險嘅單一動作（抓取一個 URL、做一次 diff），但要長時間累積先有用。Cowork 嘅優勢就喺度——人手做兩個月會放棄，自動化做兩年都唔會攰。但 5 個對手嘅月度 strategic memo，你一定要自己讀、自己諗。AI 出趨勢，你出判斷，呢個分工先講得通。

最後一個提醒：呢個流程嘅道德界線好清楚——讀公開頁面 OK、爬登入後內容唔得。Cowork 唔會自動越過呢條線，但你 prompt 入面都唔好試引佢過去。市場研究做得長唔長久，就靠你守住呢條線。
