## 情境

你 Claude Code 用咗 1 個月。每日流程：

- 「請 review my changes」（5 次 / 日）
- 「請寫 commit message」（3 次 / 日）
- 「請 執行 個 tests」（2 次 / 日）
- 「請 explain 呢段 code」（4 次 / 日）
- 「請 update CHANGELOG」（1 次 / 日）

15 個每日 prompt，每個你打一遍。計過：每日 5-10 分鐘打同樣嘢。每月 2.5-5 個鐘。

Slash command 一個字解決：
- 打 `/commit` → Claude 即時覆檢 + 寫 commit message
- 打 `/explain` → 簡短解釋揀咗嘅 code
- 打 `/test` → 跑 tests

呢類預定指令慳每日打字時間，更重要係 —— **跨 session / team 標準化 workflow**。

呢篇拆 built-in 10 個必備指令 + 教你 15 分鐘 build 自己嘅 `/command`。

## 跟住做

### 1. Built-in slash commands

開 Claude Code 後打 `/` 顯示 available commands：

**🛠 設定 + 資訊**：

- `/help` —— 顯示指令列表 + 簡介
- `/model` —— 即場切換模型（Opus / Sonnet / Haiku）
- `/config` —— 打開設定檔
- `/permissions` —— 管理工具權限
- `/cost` —— 顯示當前 session 嘅開支

**📁 處理檔案**：

- `/init` —— 將當前 directory 初始化做 Claude Code project（建立 `.claude/`）
- `/clear` —— 清除對話 context（重新開始）
- `/compact` —— 壓縮舊對話慳 context

**🔍 理解程式碼**：

- `/review` —— Review 當前 diff 揾 bug / 改進
- `/explain` —— 解釋揀咗嘅檔案 / function
- `/test` —— 跑 project tests
- `/commit` —— 由 diff 出 commit message

**🤖 Agent 管理**：

- `/agents` —— 列出 + 管理 subagents
- `/loop` —— 設定定期任務
- `/schedule` —— 排程遙距任務

⚠️ 具體 available commands 會演變。`/help` 永遠顯示當前完整列表。

### 2. 每日 workflow 例子

**朝早重設**：
```
/clear
# Fresh conversation start
```

**Session 中段 compact**（對話感覺慢）：
```
/compact
# Old messages summarized, context 保留核心
```

**Feature 完成時 commit**：
```
/review
# Claude reviews diff, flags issues
# If OK:
/commit
# Generates message, you confirm
```

**Code 解釋**（喺 IDE highlight 一個 function）：
```
/explain
# 揀選嘅 code 會被 explain
```

### 3. 點 build 自訂 slash command

每個用家都可以加自訂 `/command`。位置：

- **User-level**（你所有 project 共用）：`~/.claude/commands/`
- **Project-level**（淨係呢個 project）：`./.claude/commands/`

例：build `/release-notes` command。

建立 file `~/.claude/commands/release-notes.md`：

```markdown
---
description: Generate release notes from recent commits
---

請 review 過去 7 日嘅 git commits + generate user-facing release notes:

1. Run `git log --since="7 days ago" --oneline` 拎 commits
2. Group commits by type (feat / fix / refactor)
3. Translate technical commit message → user-readable feature description
4. Format as markdown 「## What's new」 section
5. 排序: new feature first, bug fix second
6. 唔包含 internal refactor (除非 user-facing impact)

Output structure:
## ✨ New features
- ...

## 🐛 Bug fixes
- ...

## 🔧 Improvements
- ...

語氣：concise, user-focused, 唔好 technical jargon.
```

存 file。返 Claude Code，打 `/release-notes`。

Claude 即時讀你嘅指示 file + 執行。每次 release ready → 1 個字指令 → notes 出。

### 4. 進階：command 加 arguments

自訂 command 可以接受 argument：

`~/.claude/commands/refactor-function.md`：

```markdown
---
description: Refactor a specific function with clear input
---

請 refactor 我指 function:

Function name (from 我 first message): $ARGUMENTS

Step:
1. Locate function 喺 codebase
2. Identify: complexity / readability issue
3. 提 2-3 refactor option
4. Implement after my approval
```

叫起：
```
/refactor-function calculateTotal
```

Claude 讀指示 + `$ARGUMENTS` = `calculateTotal`，知道 refactor 邊個 function。

### 5. 進階：限制 command 用嘅工具

部分 command 想限定特定工具：

`~/.claude/commands/read-only-review.md`：

```markdown
---
description: Read-only architecture review (no edits)
allowed-tools: Read, Grep, Bash(git diff:*)
---

請讀過 codebase + 出 architecture overview:

1. 用 Read tool 識 main entry point
2. Grep 揾 critical module
3. 不要 edit 任何 file
4. Output:
   - Architecture diagram (mermaid)
   - 3 個 strength
   - 3 個 weakness / refactor opportunity
```

呢類唯讀 command 安全，唔會不小心搞破壞。

### 6. 同 team 分享 commands

如果整個開發 team 用 Claude Code，將共用嘅 `./.claude/commands/` commit 入 git repo：

```bash
# In project root
mkdir -p .claude/commands
git add .claude/commands/
```

Team commit 共用 commands。新同事 pull repo + `/your-command` 即刻 work。標準化 team workflow。

## 變化

### 變化 1：將常見規律整理成 commands

用 Claude Code 1 個禮拜之後，自己 audit：

「邊個 prompt 我打過 5 次以上？」

頭 5 名 —— 每個變自訂 slash command：

- `/morning-standup` —— review 最近 commit + 起今日 standup
- `/deploy-check` —— deploy 前 checklist（tests 過 / 冇 debug code / 等）
- `/changelog-update` —— 由新 commit update CHANGELOG.md
- `/explain-error` —— 貼 error → Claude 解釋 + 提解法
- `/onboard-newhire` —— 為新 dev 出 onboarding 文檔

每個 15 分鐘 build，之後每日每個慳 5-10 分鐘。

### 變化 2：Project-level 同 user-level 嘅分別

**User-level**（`~/.claude/commands/`）：跨 project 都有用嘅通用
- /commit、/explain、/test（universal）

**Project-level**（`./.claude/commands/`）：呢個 codebase 獨有
- /deploy-to-vercel（Next.js project）
- /generate-migration（Supabase project）
- /publish-to-pypi（Python lib project）

Project-level command 跟住 codebase 走 —— team 入同一個 repo 用同一啲 commands。

### 變化 3：Slash commands + skills + hooks 三者組合

3 層自訂 stack：

- **Skills**（按第 79 篇）：重複 workflow 加規則（例如 email-polish-hk skill）
- **Slash commands**（呢篇）：明確觸發嘅命名程序
- **Hooks**（第 87 篇）：事件觸發嘅自動化（例如存檔時自動排版）

用途有時重疊 —— 你揀邊一層：

- 「我想明確觸發」→ slash command
- 「我想 Claude 自己決定叫起」→ skill
- 「我想系統事件自動觸發」→ hook

例：
- `/format-code` slash command（明確、按需）
- email-polish skill（自動 detect 我貼 email）
- 存檔 hook（自動跑 linter）

## 一個心態

Slash command 嘅深層價值唔係「打字速度」—— 係**將個人 workflow 固化做可重用嘅命名程序**。

你個腦每日重新發明「點 commit」「點 release notes」「點解釋 bug」。每次重新發明 5-10 分鐘心理負擔。Slash command = 1 次設計 + 重複執行。

長遠：你嘅 `~/.claude/commands/` 變成你嘅「工程思維延伸」—— 10-20 個自訂指令多年磨利。新同事睇到你 commands 就明你 workflow 哲學。

最後提醒：

- ✅ **第 1 個自訂 command 唔需要完美**。15 分鐘 ship，每星期調整。
- ⚠️ **唔好預先 over-engineer command**。確認你真係每日用嘅規律先 codify。
- 🎯 **將 team 層 command commit 入 git repo**。係 onboarding 工具 + workflow 標準化。

下個鐘 build 1 個自訂 command。1 個禮拜後你會本能咁打 `/your-command`。
