## 情境

你 Claude Code 用 1 個月。每日 daily flow:

- 「請 review my changes」(5 次 / day)
- 「請寫 commit message」(3 次 / day)
- 「請 run 個 tests」(2 次 / day)
- 「請 explain 呢段 code」(4 次 / day)
- 「請 update CHANGELOG」(1 次 / day)

15 個 daily prompt，每個你 type 一遍。計過：daily 5-10 分鐘 typing 同樣嘢。每月 2.5-5 hr。

Slash command 一個 word 解決：
- Type `/commit` → Claude 即時 review + 寫 commit message
- Type `/explain` → 簡 explain selected code
- Type `/test` → run tests

呢類預定 command 慳 daily typing 時間，更重要 — **標準化 workflow** across sessions / team members。

呢篇拆 built-in 10 個必備 commands + 教你 build own custom `/command` 15 分鐘。

## 跟住做

### 1. Built-in slash commands

開 Claude Code 後 type `/` 顯示 available commands:

**🛠 Setup + info**:

- `/help` — Show commands list + brief description
- `/model` — Switch model (Opus / Sonnet / Haiku) inline
- `/config` — Open settings.json
- `/permissions` — Manage tool permissions
- `/cost` — Show current session API cost

**📁 Working with files**:

- `/init` — Initialize current directory as Claude Code project (create `.claude/`)
- `/clear` — Clear conversation context (start fresh)
- `/compact` — Compress old conversation to save context window

**🔍 Code understanding**:

- `/review` — Review current diff for bugs / improvements
- `/explain` — Explain selected file / function
- `/test` — Run project tests
- `/commit` — Generate commit message from diff

**🤖 Agent management**:

- `/agents` — List + manage subagents
- `/loop` — Set up recurring task
- `/schedule` — Schedule remote task

⚠️ 具體 available commands 會演變。`/help` 永遠顯示 current full list。

### 2. Daily workflow examples

**朝早 daily reset**:
```
/clear
# Fresh conversation start
```

**Mid-session compact** (conversation 感覺慢):
```
/compact
# Old messages summarized, context 保留核心
```

**End-of-feature commit**:
```
/review
# Claude reviews diff, flags issues
# If OK:
/commit
# Generates message, you confirm
```

**Code 解釋** (highlight a function in IDE):
```
/explain
# 揀選嘅 code 會被 explain
```

### 3. 點 build custom slash command

每個 user 都可以加 custom `/command`。位置:

- **User-level** (你所有 project share): `~/.claude/commands/`
- **Project-level** (此 project only): `./.claude/commands/`

例：build `/release-notes` command。

Create file `~/.claude/commands/release-notes.md`:

```markdown
---
description: Generate release notes from recent commits
---

請 review 過去 7 日嘅 git commits + generate user-facing release notes:

1. Run `git log --since="7 days ago" --oneline` 拎 commits
2. Group commits by type (feat / fix / refactor)
3. Translate technical commit message → user-readable feature description
4. Format as markdown 「## What's new」 section
5. 排序: new feature first, bug fix second
6. 唔包含 internal refactor (除非 user-facing impact)

Output structure:
## ✨ New features
- ...

## 🐛 Bug fixes
- ...

## 🔧 Improvements
- ...

語氣：concise, user-focused, 唔好 technical jargon.
```

Save file。返 Claude Code，type `/release-notes`。

Claude 即 read 你 instruction file + execute。每次 release ready → 1 個 word command → notes 出。

### 4. 進階：command with arguments

Custom command 可以接受 argument:

`~/.claude/commands/refactor-function.md`:

```markdown
---
description: Refactor a specific function with clear input
---

請 refactor 我指 function:

Function name (from 我 first message): $ARGUMENTS

Step:
1. Locate function 喺 codebase
2. Identify: complexity / readability issue
3. 提 2-3 refactor option
4. Implement after my approval
```

Invoke:
```
/refactor-function calculateTotal
```

Claude read instruction + `$ARGUMENTS` = `calculateTotal`，知道 refactor 邊個 function。

### 5. 進階：tool restriction in command

部分 command 想限制特定 tools:

`~/.claude/commands/read-only-review.md`:

```markdown
---
description: Read-only architecture review (no edits)
allowed-tools: Read, Grep, Bash(git diff:*)
---

請讀過 codebase + 出 architecture overview:

1. 用 Read tool 識 main entry point
2. Grep 揾 critical module
3. 不要 edit 任何 file
4. Output:
   - Architecture diagram (mermaid)
   - 3 個 strength
   - 3 個 weakness / refactor opportunity
```

呢類 read-only command 安全，唔會不小心搞破壞嘢。

### 6. 同 team 分享 commands

如整個 dev team 用 Claude Code，shared `./.claude/commands/` 入 git repo:

```bash
# In project root
mkdir -p .claude/commands
git add .claude/commands/
```

Team commit shared commands。新 hire pull repo + `/your-command` works immediately。標準化 team workflow。

## 變化

### 變化 1：將常見 patterns 整理 commands list

用 Claude Code 1 個禮拜後，audit 自己:

「邊個 prompt 我 type > 5 次?」

Top 5 list — 每個變 custom slash command:

- `/morning-standup` — review last commit + draft today's standup
- `/deploy-check` — pre-deploy checklist (tests passed / no debug code / etc)
- `/changelog-update` — update CHANGELOG.md from new commit
- `/explain-error` — paste error → Claude explain + suggest fix
- `/onboard-newhire` — generate onboarding doc for new dev

每個 15 分鐘 build，後續 daily 慳 5-10 分鐘每個。

### 變化 2：Project-specific 同 user-level 嘅分別

**User-level** (`~/.claude/commands/`): 跨 project 都有用嘅 generic
- /commit, /explain, /test (universal)

**Project-level** (`./.claude/commands/`): 呢個 codebase 獨有
- /deploy-to-vercel (Next.js project)
- /generate-migration (Supabase project)
- /publish-to-pypi (Python lib project)

Project-level command 跟 codebase 走 — team 入 same repo 用 same commands。

### 變化 3：Slash commands + skills + hooks 三者組合

3-tier customization stack:

- **Skills** (per 第 79 篇): 重複 workflow with rules (e.g., email-polish-hk skill)
- **Slash commands** (此篇): 明確 trigger 嘅 named procedure
- **Hooks** (第 87 篇): event-triggered 自動 (e.g., auto-format on save)

Use case 有時 overlap — 你揀邊個 tier:

- 「我想明確 trigger」→ slash command
- 「我想 Claude 自己決定 invoke」→ skill
- 「我想 system event 自動 trigger」→ hook

例:
- `/format-code` slash command (明確, on-demand)
- email-polish skill (自動 detect 我 paste email)
- on-save hook (自動 run linter)

## 一個 mindset

Slash command 嘅深層價值唔係「typing speed」— 係 **將個人 workflow 固化做 reusable 命名程序**。

你個腦每日 reinvent「點 commit」「點 release notes」「點 explain bug」。每次 reinvent 5-10 分鐘 mental load。Slash command = 1 次 design + 重複 execution。

長遠：你嘅 `~/.claude/commands/` 變成你嘅「工程思維延伸」— 10-20 個 custom commands 多年磨利。新 hire 睇到你 commands 就明你 workflow philosophy。

最後 reminder：

- ✅ **第 1 個 custom command 唔需要完美**。15 分鐘 ship，每週 iterate。
- ⚠️ **唔好 over-engineer command upfront**。確認你真係 daily use 嘅 pattern 先 codify。
- 🎯 **Share team-level command 入 git repo**。係 onboarding 工具 + workflow 標準化。

下個鐘 build 1 個 custom command。1 個禮拜後你會 instinctively type `/your-command` daily。
